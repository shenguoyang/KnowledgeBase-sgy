# 文章生成提示词
## 主题
以XXX模式为例写一篇知识分享类文章

## 具体要求
### 1. 文章开头
+ 必须加上目录

### 2. 未使用设计模式的代码示例与问题分析
+ **代码场景**：描述该代码的模拟业务场景
+ **代码实现**：给出存在潜在问题的具体 C++ 代码（要求客户端调用也附上）
+ **问题分析角度**（不局限于以下角度）：
    - 扩展性问题：增加新功能时需要修改多处代码
    - 耦合性问题：模块/类之间依赖过紧
    - 复用性问题：相似代码重复出现
    - 维护性问题：需求变更导致大规模重构
    - 性能问题：不必要的对象创建或重复计算

           ** **特别注意****

            如果未使用设计模式的代码没有耦合性问题，则不要输出耦合性问题小节。如果有则输出。

+ **重点说明**：这些缺陷在实际项目演进中可能带来的具体后果

### 3. 引出设计模式
+ 引出本次需要讲解的设计模式
+ 说明该设计模式提出的灵感

### 4. 应用设计模式的解决方案
+ **重构后的代码实现**：展示使用设计模式后的 C++ 代码（要求客户端调用也附上）
+ **解决方案分析角度**（不局限于以下角度）：
    - 如何解耦对象关系（用组合代替继承等）
    - 如何提高扩展性（通过接口隔离变化点等）
    - 如何增强复用性（模板方法/对象池等等）
    - 如何提升维护性（符合开闭原则等）
    - 性能优化效果（减少资源消耗等）

           ** **特别注意****

            如果应用设计模式的代码没有以上的优点，请不要强行说有。例如没有对维护性有提高，则不输出如何提升维护性小节内容。

+ **重点对比**：通过代码差异对比说明改进原理

### 5. 设计模式核心总结
+ **核心思想**：用一句话概括模式的本质
+ **UML类图**：标准模式类图（标注 C++ 实现特性）
+ **各角色职责说明**：用 C++ 术语描述
+ **典型应用场景**：
    - 适合的业务场景（结合 C++ 特性说明）
    - 反例场景（说明不适用的情况）

### 6. 思考问题
+ 给出留给读者思考的问题

### 7. 输出格式
+ 整篇文章以 Markdown 格式输出
+ 排版美观

带迭代工作流的设计模式知识分享文章生成提示词  
整体生成工作流（强制6步迭代流程）  
全文严格按照以下六步工作流递进创作，实现「基础场景-原始代码-问题剖析-标准模式解决-需求迭代-变种模式解决」的完整项目演进逻辑，区别于传统单一模式讲解，贴合真实项目迭代流程：  
第一步：基础需求场景描述（锁定初始稳定业务需求，无迭代变更）  
第二步：生成未应用设计模式的原始代码（基于基础场景编写有缺陷的C++代码+客户端调用）  
第三步：深度剖析原始代码存在的核心问题（从多维度分析缺陷+项目演进实际后果）  
第四步：应用标准设计模式重构解决问题（标准模式代码重构+方案优势分析+新旧代码对比）  
第五步：业务需求场景变更迭代（基于基础场景提出合理的新增/修改需求，制造新的适配痛点）  
第六步：应用变种设计模式解决迭代后需求（基于需求变更，使用对应模式的变种写法重构代码，解决新痛点，完成技术迭代）

文章核心创作要求（保留原有全部规则，适配新增工作流）  
主题  
以XXX模式为例写一篇知识分享类技术科普文章  
具体格式与内容要求

1. 文章开头  
● 必须加上全自动生成文章目录，涵盖工作流6大步骤及所有核心小节
2. 基础场景+未使用设计模式的代码示例与问题分析（对应工作流1-3步）  
● 业务场景：详细描述初始版本的模拟业务场景，需求清晰、贴合C++后端/客户端真实开发场景  
● 代码实现：给出存在潜在问题的完整C++代码，必须附带完整客户端调用代码，代码可直接编译运行  
● 问题分析角度（按需输出，无对应问题则删除对应小节，禁止强行堆砌）：
+ 扩展性问题：增加新功能时需要修改多处代码
+ 耦合性问题：模块/类之间依赖过紧
+ 复用性问题：相似代码重复出现
+ 维护性问题：需求变更导致大规模重构
+ 性能问题：不必要的对象创建或重复计算  
● 重点说明：结合企业级项目演进，详细阐述以上缺陷在版本迭代、多人协作、需求更新中可能带来的具体实际后果（BUG风险、重构成本、维护难度、扩展性瓶颈等）
3. 引出标准设计模式  
● 结合原始代码痛点，自然引出本次需要讲解的标准原版设计模式  
● 说明该标准设计模式的设计灵感、核心解决的通用技术痛点
4. 应用标准设计模式的解决方案（对应工作流第4步）  
● 重构后的代码实现：展示使用标准原版设计模式重构后的完整C++代码，附带客户端调用代码  
● 解决方案分析角度（按需输出，无对应优势则删除小节，禁止虚假吹捧）：
+ 如何解耦对象关系（用组合代替继承等）
+ 如何提高扩展性（通过接口隔离变化点等）
+ 如何增强复用性（模板方法/对象池等等）
+ 如何提升维护性（符合开闭原则等）
+ 性能优化效果（减少资源消耗等）  
● 重点对比：通过新旧代码差异逐点对比，清晰说明标准模式的改进原理与设计思路
5. 业务需求迭代与新模式痛点（对应工作流第5步）  
● 基于初始业务场景，提出合理的需求变更/功能扩展（贴合真实业务迭代逻辑）  
● 分析：标准设计模式在新需求场景下的局限性、适配短板，引出需要使用变种模式的原因
6. 应用变种设计模式的迭代解决方案（对应工作流第6步）  
● 明确说明当前使用的设计模式变种版本，区分与标准模式的差异  
● 重构后的代码实现：展示使用变种设计模式适配迭代需求的完整C++代码，附带客户端调用  
● 解决方案分析：结合迭代需求，分析变种模式的优化点、适配优势、解决的新痛点  
● 对比总结：标准模式 vs 变种模式 的适用场景、优缺点、迭代改进逻辑
7. 设计模式核心总结（含标准+变种模式）  
● 核心思想：分别用一句话概括标准模式、变种模式的本质  
● UML类图：输出标准模式标准UML类图、变种模式优化后UML类图，标注C++实现特性  
● 各角色职责说明：使用C++专业术语描述标准、变种模式下各角色职责变化  
● 典型应用场景：
+ 适合的业务场景（结合C++特性分别说明标准/变种模式适用场景）
+ 反例场景（说明两种模式不适用的业务场景与技术场景）
8. 思考问题  
● 结合「基础场景-标准模式-需求迭代-变种模式」全流程，给出2-3个贴合实战、引发深度思考的读者思考题
9. 输出格式要求  
● 整篇文章以标准Markdown格式输出，层级清晰、排版美观、代码块高亮、段落分区明确  
● 代码统一使用C++代码块标识，注释清晰、可直接编译运行  
● 严格遵循6步工作流逻辑递进，全文逻辑闭环，从原始问题到最终迭代解决方案完整贯穿
