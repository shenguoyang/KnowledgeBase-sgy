### <font style="color:rgb(15, 17, 21);">综合示例：电商的“优惠券”功能演进</font>
**<font style="color:rgb(15, 17, 21);">场景：</font>**<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">产品经理提出：“我们需要升级优惠券系统，支持一种新券：用户领取后，分享给朋友，朋友使用后双方都能获得奖励。”</font>

+ **<font style="color:rgb(15, 17, 21);">传统模式可能的结果：</font>**
    - <font style="color:rgb(15, 17, 21);">产品写一份冗长的PRD。</font>
    - <font style="color:rgb(15, 17, 21);">开发看到后，直接在原有的</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">Coupon</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">表里加几个字段（如</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">is_shareable</font>`<font style="color:rgb(15, 17, 21);">,</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">sharer_benefit</font>`<font style="color:rgb(15, 17, 21);">），然后修改“领券”、“用券”的Service方法，加入一堆</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">if-else</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">判断。</font>
    - **<font style="color:rgb(15, 17, 21);">问题：</font>**<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">业务逻辑和技术实现高度纠缠。下次再想增加“过期自动提醒”或“基于用户画像的精准发放”，代码会越来越混乱。</font>
+ **<font style="color:rgb(15, 17, 21);">DDD模式下的协作过程：</font>**
    1. **<font style="color:rgb(15, 17, 21);">事件风暴（Event Storming）工作坊：</font>**<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">产品、开发、测试一起，用贴纸梳理业务事件。</font>
        * <font style="color:rgb(15, 17, 21);">产品经理主导描述：“</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">用户领取了分享券</font>`<font style="color:rgb(15, 17, 21);">” -> “</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">用户生成了分享链接</font>`<font style="color:rgb(15, 17, 21);">” -> “</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">好友通过链接领取了券</font>`<font style="color:rgb(15, 17, 21);">” -> “</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">好友使用了分享券</font>`<font style="color:rgb(15, 17, 21);">” -> “</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享者和使用者均获得了奖励</font>`<font style="color:rgb(15, 17, 21);">”。</font>
    2. **<font style="color:rgb(15, 17, 21);">建模与通用语言定义：</font>**
        * <font style="color:rgb(15, 17, 21);">大家发现，原有的“优惠券”模型无法承载新业务。于是识别出两个新的核心概念：</font>`**<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享券（ShareableCoupon）</font>**`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">和</font><font style="color:rgb(15, 17, 21);"> </font>`**<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">奖励规则（RewardRule）</font>**`<font style="color:rgb(15, 17, 21);">。</font>
        * <font style="color:rgb(15, 17, 21);">明确通用语言：</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享券</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">是一种特殊的</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">优惠券</font>`<font style="color:rgb(15, 17, 21);">，它有</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享人（sharer）</font>`<font style="color:rgb(15, 17, 21);">、</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">状态（已领取/已分享/已核销）</font>`<font style="color:rgb(15, 17, 21);">。</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">奖励规则</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">定义了当</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享券</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">被核销时，如何给</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享人</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">和</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">使用人</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">发放</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">奖励（可能是积分或新券）</font>`<font style="color:rgb(15, 17, 21);">。</font>
    3. **<font style="color:rgb(15, 17, 21);">战略设计与解耦：</font>**
        * <font style="color:rgb(15, 17, 21);">团队分析认为，“优惠券的发放和核销”与“分享与奖励的追踪”是不同密度的业务。决定将其拆分到两个限界上下文：</font>
            + **<font style="color:rgb(15, 17, 21);">「营销券」上下文：</font>**<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">负责券的创建、领取、核销等基本生命周期。它只需要知道一张券是不是“分享券”，以及核销时触发一个“分享券已核销”的领域事件。</font>
            + **<font style="color:rgb(15, 17, 21);">「社交推广」上下文：</font>**<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">订阅“分享券已核销”事件。它维护</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">分享关系</font>`<font style="color:rgb(15, 17, 21);">和</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">奖励规则</font>`<font style="color:rgb(15, 17, 21);">，在收到事件后，执行复杂的奖励发放逻辑。</font>
    4. **<font style="color:rgb(15, 17, 21);">技术实现：</font>**
        * <font style="color:rgb(15, 17, 21);">开发人员根据以上模型，在「营销券」上下文中新增</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">ShareableCoupon</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">类，继承或组合原有的</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">Coupon</font>`<font style="color:rgb(15, 17, 21);">。核销后发布</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">ShareableCouponUsedEvent</font>`<font style="color:rgb(15, 17, 21);">。</font>
        * <font style="color:rgb(15, 17, 21);">在「社交推广」上下文中，开发</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">RewardService</font>`<font style="color:rgb(15, 17, 21);">，监听上述事件，并调用</font><font style="color:rgb(15, 17, 21);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">RewardRule</font>`<font style="color:rgb(15, 17, 21);"> </font><font style="color:rgb(15, 17, 21);">计算并发放奖励。</font>
        * **<font style="color:rgb(15, 17, 21);">解耦体现：</font>**<font style="color:rgb(15, 17, 21);"> 未来如果奖励规则变得极其复杂（比如引入阶梯奖励、延迟到账），只需要修改「社交推广」上下文，完全不会触动「营销券」的核心核销流程。</font>

我们以电商订单场景为例，拆解 DDD 中领域事件、聚合根、实体、值对象、仓储的概念，并基于 C++ 实现核心代码。

场景定义

用户下单时，创建一个订单聚合：

+ 聚合根： Order （订单），作为聚合的入口，管控整个聚合的一致性  

+ 实体： OrderItem （订单项），有唯一标识，随订单一起存在  

+ 值对象： Address （收货地址）、 Money （金额），无唯一标识，属性不可变  

+ 领域事件： OrderCreatedEvent （订单创建事件），订单创建后发布，用于通知库存扣减、消息推送等下游操作  

+ 仓储： IOrderRepository （订单仓储接口），负责聚合根的持久化与查询，屏蔽底层数据库细节



代码核心说明

1. 值对象： Money 、 Address  无唯一标识，通过属性判断相等，且属性只读，保证不可变性。  
  
2. 实体： OrderItem  有唯一  item_id ，是聚合内的可变对象，只能通过聚合根操作。  
可以分为三类实体：聚合根实体；聚合内实体；独立实体  
3. 聚合根： Order  是聚合的唯一入口，所有对订单项的修改都需通过  addOrderItem  方法，确保业务规则不被破坏。  
  
4. 领域事件： OrderCreatedEvent  在订单确认创建时发布，用于解耦订单创建与下游业务。  
  
5. 仓储： IOrderRepository  只操作聚合根  Order ，屏蔽了数据库的具体实现， OrderRepositoryImpl  是其具体落地。
