### 代码坏味道：**发散式变化（Divergent Change）**
#### 定义
发散式变化是指一个类因为**不同的原因**而需要频繁修改。换句话说，当需求变化时，你需要在同一个类的多个地方进行修改，而不是集中在一个地方。这种坏味道通常表明类的职责不够单一，违反了**单一职责原则（SRP）**。

#### 示例场景
假设我们有一个 `Report` 类，它负责生成报告并将其保存为不同的格式（如 `PDF` 和 `CSV`）。随着需求的变化，我们可能需要修改报告的生成逻辑或保存逻辑，而这些修改通常发生在同一个类中。

---

### 重构前代码
cpp

复制

```plain
#include <iostream>
#include <string>

// 报告类（违反单一职责原则）
class Report {
public:
    void generateReport() {
        // 生成报告的逻辑
        std::cout << "Generating report content..." << std::endl;
    }

    void saveToPdf(const std::string& filePath) {
        // 保存为 PDF 的逻辑
        std::cout << "Saving report to PDF: " << filePath << std::endl;
    }

    void saveToCsv(const std::string& filePath) {
        // 保存为 CSV 的逻辑
        std::cout << "Saving report to CSV: " << filePath << std::endl;
    }
};

// 客户端代码
int main() {
    Report report;
    report.generateReport();
    report.saveToPdf("report.pdf");
    report.saveToCsv("report.csv");

    return 0;
}
```

---

### 问题分析
1. **发散式变化**：
    - 如果报告生成逻辑需要修改（例如添加新的数据字段），我们需要修改 `generateReport()` 方法。
    - 如果保存逻辑需要修改（例如支持新的格式），我们需要修改 `saveToPdf()` 或 `saveToCsv()` 方法。
    - 这些修改的原因不同，但都集中在同一个类中，导致类不稳定。
2. **违反单一职责原则**：
    - `Report` 类同时负责生成报告和保存报告，职责过多。

---

### 重构思路
1. **分离职责**：
    - 将报告生成和保存的逻辑分离到不同的类中。
    - 使用策略模式（Strategy Pattern）来封装不同的保存逻辑。
2. **引入接口**：
    - 定义一个 `ReportSaver` 接口，用于统一保存行为。
    - 具体的保存逻辑由实现类（如 `PdfSaver` 和 `CsvSaver`）完成。

---

### 重构后代码
cpp

复制

```plain
#include <iostream>
#include <string>
#include <memory>

// 报告保存接口
class ReportSaver {
public:
    virtual void save(const std::string& filePath) = 0;
    virtual ~ReportSaver() = default;
};

// 具体保存类：PDF
class PdfSaver : public ReportSaver {
public:
    void save(const std::string& filePath) override {
        std::cout << "Saving report to PDF: " << filePath << std::endl;
    }
};

// 具体保存类：CSV
class CsvSaver : public ReportSaver {
public:
    void save(const std::string& filePath) override {
        std::cout << "Saving report to CSV: " << filePath << std::endl;
    }
};

// 报告类（职责单一）
class Report {
public:
    void generateReport() {
        // 生成报告的逻辑
        std::cout << "Generating report content..." << std::endl;
    }

    void saveReport(std::unique_ptr<ReportSaver> saver, const std::string& filePath) {
        saver->save(filePath);
    }
};

// 客户端代码
int main() {
    Report report;
    report.generateReport();

    // 保存为 PDF
    report.saveReport(std::make_unique<PdfSaver>(), "report.pdf");

    // 保存为 CSV
    report.saveReport(std::make_unique<CsvSaver>(), "report.csv");

    return 0;
}
```

---

### 重构后优点
1. **职责分离**：
    - `Report` 类只负责生成报告，保存逻辑由 `ReportSaver` 及其子类负责。
    - 符合单一职责原则，类的职责更加清晰。
2. **扩展性增强**：
    - 如果需要支持新的保存格式（如 `Excel`），只需新增一个 `ExcelSaver` 类，无需修改 `Report` 类。
    - 符合开闭原则（对扩展开放，对修改关闭）。
3. **减少发散式变化**：
    - 修改报告生成逻辑时，只需修改 `Report` 类。
    - 修改保存逻辑时，只需修改具体的 `ReportSaver` 实现类。

### 总结
通过将 **发散式变化** 的代码重构为职责单一、结构清晰的类，我们不仅解决了代码坏味道，还提高了代码的可维护性和扩展性。这种重构方法在大型项目中尤为重要，能够有效降低代码的复杂性和耦合度。
