### 代码坏味道：**基本类型偏执（Primitive Obsession）**
#### 定义
**基本类型偏执** 是指过度使用基本数据类型（如 `int`、`string`、`double`）来表示业务概念，而不是将其封装为专门的类。这种坏味道会导致：

+ **重复校验逻辑**：同一数据规则分散在多处。
+ **可读性差**：代码无法直接体现业务含义（例如用 `string` 表示电话号码）。
+ **类型不安全**：编译器无法检查业务规则的合法性（例如货币单位不匹配）。

---

### 重构前示例：用 `string` 表示电话号码
以下代码用 `string` 直接存储电话号码，校验逻辑缺失或重复。

```cpp
#include <iostream>
#include <string>

// 客户类（直接用 string 存储电话号码）
class Customer {
public:
    Customer(const std::string& name, const std::string& phone) 
        : name_(name), phone_(phone) {}

    void setPhone(const std::string& phone) {
        // 未校验电话号码格式
        phone_ = phone;
    }

    void printInfo() {
        std::cout << "Name: " << name_ << ", Phone: " << phone_ << std::endl;
    }

private:
    std::string name_;
    std::string phone_; // 🟡 基本类型偏执
};

// 客户端代码
int main() {
    Customer customer("Alice", "123-456"); // 无效电话号码
    customer.printInfo();

    customer.setPhone("+1 650-1234567");   // 格式正确，但未校验
    customer.printInfo();

    return 0;
}
```

#### 问题分析
1. **无效数据**：电话号码格式未校验，可能存储非法值（如 `"123-456"`）。
2. **重复校验**：若其他类也需处理电话号码，校验逻辑需重复实现。
3. **缺乏业务语义**：`string` 类型无法直接表达“电话号码”这一业务概念。

---

### 重构思路
1. **封装业务概念**：创建 `PhoneNumber` 类，封装电话号码的存储和校验逻辑。
2. **集中校验规则**：在构造函数和 `setter` 中强制校验格式。
3. **增强类型安全**：通过自定义类型避免错误赋值。

---

### 重构后代码
```cpp
#include <iostream>
#include <string>
#include <stdexcept>
#include <regex>

// ----------------------
// 封装电话号码的专用类
// ----------------------
class PhoneNumber {
public:
    PhoneNumber(const std::string& number) {
        if (!isValid(number)) {
            throw std::invalid_argument("Invalid phone number format.");
        }
        number_ = number;
    }

    std::string getNumber() const { 
        return number_; 
    }

    // 可选：提供格式化输出
    std::string format() const {
        return "Tel: " + number_;
    }

private:
    std::string number_;

    // 集中校验逻辑（正则表达式示例）
    static bool isValid(const std::string& number) {
        std::regex pattern(R"(\+\d{1,3} \d{3}-\d{6,7})"); // 示例：+国家代码 区号-号码
        return std::regex_match(number, pattern);
    }
};

// ----------------------
// 重构后的客户类
// ----------------------
class Customer {
public:
    Customer(const std::string& name, const PhoneNumber& phone)
        : name_(name), phone_(phone) {}

    void setPhone(const PhoneNumber& phone) {
        phone_ = phone; // 安全赋值（已通过构造函数校验）
    }

    void printInfo() const {
        std::cout << "Name: " << name_ 
                  << ", " << phone_.format() << std::endl;
    }

private:
    std::string name_;
    PhoneNumber phone_; // 🟢 使用专用类
};

// 客户端代码
int main() {
    try {
        PhoneNumber phone("+1 650-1234567"); // 合法格式
        Customer customer("Alice", phone);
        customer.printInfo();

        // 尝试非法电话号码
        // PhoneNumber invalidPhone("123-456"); // 抛出异常
    } catch (const std::exception& e) {
        std::cerr << "Error: " << e.what() << std::endl;
    }

    return 0;
}
```

---

### 重构后优点
1. **业务语义明确**：
    - `PhoneNumber` 类直接体现业务概念，代码可读性提升。
2. **集中校验逻辑**：
    - 电话号码格式校验在 `PhoneNumber` 类中统一实现，避免重复。
3. **类型安全**：
    - 无法直接将 `string` 赋值给 `PhoneNumber`，必须通过构造函数（隐式校验）。
4. **易于扩展**：
    - 添加新功能（如格式化、区号提取）只需修改 `PhoneNumber` 类。

### 另一个示例：用 `double` 表示温度范围
假设用两个 `double` 表示温度范围，重构为 `TemperatureRange` 类：

```cpp
// 重构前
double minTemp = -10.0;
double maxTemp = 40.0;

// 重构后
class TemperatureRange {
public:
    TemperatureRange(double min, double max) : min_(min), max_(max) {
        if (min > max) throw std::invalid_argument("Invalid range.");
    }

    double getMin() const { return min_; }
    double getMax() const { return max_; }

    bool contains(double temp) const {
        return temp >= min_ && temp <= max_;
    }

private:
    double min_;
    double max_;
};
```

---

### 何时应该重构基本类型偏执？
1. **数据有明确业务含义**：如金额、坐标、日期等。
2. **需要重复校验或格式化**：如电话号码、邮箱地址。
3. **需要封装行为**：如温度范围是否包含某个值。

---

### 总结
**基本类型偏执** 的本质是忽视了对业务概念的抽象。通过重构：

1. **提升内聚性**：数据和操作被封装在同一个类中。
2. **增强健壮性**：非法数据在构造时即被拦截。
3. **代码即文档**：类名和方法名直接传达业务语义。

这种重构是领域驱动设计（DDD）中“**值对象**”思想的直接应用，能够显著提高代码的可维护性和可靠性。
