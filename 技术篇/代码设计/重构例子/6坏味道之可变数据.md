### 代码坏味道：**可变数据（Mutable Data）**
#### 定义
**可变数据** 是指代码中允许数据被随意修改，导致状态难以追踪、副作用不可预测。这种坏味道常见于以下场景：

+ 类暴露公共可修改的成员变量。
+ 对象的状态被多个方法或外部代码频繁修改。
+ 缺乏对数据修改的约束，导致业务规则被破坏。

可变数据会引发 **竞态条件**、**逻辑错误**，并使得代码难以调试和维护。

---

### 示例场景
假设我们有一个 `Player` 类表示游戏玩家，其生命值（`health`）和位置（`x`, `y`）被直接暴露为公共变量，允许外部随意修改。这可能导致生命值被设为负数，或位置被非法设置。

---

### 重构前代码
```cpp
#include <iostream>

// 玩家类（暴露可变数据）
class Player {
public:
    int health = 100; // 🟡 公共可变数据
    int x = 0;        // 🟡
    int y = 0;        // 🟡

    void printStatus() {
        std::cout << "Health: " << health 
                  << ", Position: (" << x << ", " << y << ")\n";
    }
};

// 客户端代码
int main() {
    Player player;
    player.printStatus(); // 初始状态

    // 外部代码随意修改数据
    player.health = -50;  // ❌ 非法生命值
    player.x = 1000;      // ❌ 越界位置
    player.y = -200;

    player.printStatus(); // 输出异常状态

    return 0;
}
```

---

### 问题分析
1. **直接暴露数据**：
    - `health`、`x`、`y` 是公共变量，外部代码可以绕过业务规则直接修改。
    - 例如，生命值被设为负数，位置超出地图范围。
2. **缺乏封装**：
    - 修改数据的逻辑分散在外部代码中，无法统一校验或约束。
3. **线程不安全**：
    - 多线程环境下，公共变量可能被并发修改，导致竞态条件。

---

### 重构思路
1. **封装数据**：
    - 将成员变量设为 `private`，通过方法控制访问。
    - 添加校验逻辑，确保数据修改符合业务规则。
2. **提供安全接口**：
    - 用 `setHealth()`、`moveTo()` 等方法替代直接变量访问。
    - 支持**原子操作**（如“移动并校验位置”）。
3. **引入不可变性**：
    - 对某些数据（如玩家唯一ID）使用 `const` 成员变量。

---

### 重构后代码
```cpp
#include <iostream>
#include <stdexcept>

class Player {
public:
    Player(int initialX, int initialY) 
        : x_(initialX), y_(initialY) {}  // 位置通过构造函数初始化

    // 获取生命值（只读）
    int health() const { return health_; }

    // 安全修改生命值
    void setHealth(int value) {
        if (value < 0) {
            health_ = 0;
        } else if (value > 100) {
            health_ = 100;
        } else {
            health_ = value;
        }
    }

    // 安全移动（校验位置范围）
    void moveTo(int newX, int newY) {
        if (newX < 0 || newX > 1000 || newY < 0 || newY > 1000) {
            throw std::out_of_range("Invalid position!");
        }
        x_ = newX;
        y_ = newY;
    }

    void printStatus() const {
        std::cout << "Health: " << health_ 
                  << ", Position: (" << x_ << ", " << y_ << ")\n";
    }

private:
    int health_ = 100; // 🟢 私有变量
    const int id_ = 123; // 🟢 不可变数据（假设ID固定）
    int x_; // 🟢 通过方法控制修改
    int y_;
};

// 客户端代码
int main() {
    Player player(0, 0);
    player.printStatus();

    // 安全修改生命值
    player.setHealth(120); // 会被限制为 100
    player.setHealth(-30); // 会被限制为 0

    // 安全移动
    try {
        player.moveTo(500, 600); // 合法位置
        // player.moveTo(1500, -100); // 抛出异常
    } catch (const std::exception& e) {
        std::cerr << "Error: " << e.what() << std::endl;
    }

    player.printStatus();

    // 不可变数据示例
    // player.id_ = 456; // ❌ 编译错误（id_ 是 private const）

    return 0;
}
```

---

### 重构后优点
1. **数据受控**：
    - `health` 的取值范围被限制在 `[0, 100]`，位置移动前校验合法性。
    - 业务规则集中管理，避免数据污染。
2. **线程安全性提升**：
    - 若在方法内部添加锁（如 `std::mutex`），可安全支持多线程访问。
3. **错误早抛**：
    - 非法操作（如越界移动）直接抛出异常，避免后续逻辑错误。
4. **不可变数据**：
    - `id_` 等关键数据通过 `const` 强制不可变，防止意外修改。

---

### 输出结果
```plain
Health: 100, Position: (0, 0)
Health: 0, Position: (500, 600)
```

---

### 何时应该限制可变性？
1. **核心业务数据**：如订单金额、用户ID等。
2. **共享状态**：多线程环境下的共享变量。
3. **复杂校验逻辑**：数据修改需要满足特定条件时。

---

### 
### 总结
**可变数据** 是代码中潜在的“定时炸弹”。通过重构：

1. **封装可变性**：限制数据修改的入口。
2. **集中校验逻辑**：确保数据始终合法。
3. **拥抱不可变性**：对关键数据“上锁”。

这种重构不仅能减少 Bug，还能提高代码的可维护性和并发安全性，是构建健壮系统的基石。
