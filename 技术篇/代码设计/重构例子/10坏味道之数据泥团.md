### 代码坏味道：**数据泥团（Data Clumps）**
#### 定义
**数据泥团** 是指一组数据项（如类的字段、方法的参数等）**频繁同时出现**在多个地方，但这些数据没有被合理封装为一个独立的概念。这种坏味道会导致代码重复、可维护性降低，并可能掩盖业务逻辑中的深层概念。

---

### 示例场景
假设我们有一个电商系统，需要处理用户地址信息和商品物流信息。以下代码中，`街道`、`城市`、`邮编`这三个数据项总是成组出现，但没有被封装为一个整体。

---

### 重构前代码
```cpp
#include <iostream>
#include <string>

// 用户类（未封装地址信息）
class User {
public:
    User(const std::string& name, 
         const std::string& street, 
         const std::string& city, 
         const std::string& zipCode)
        : name_(name), street_(street), city_(city), zipCode_(zipCode) {}

    void printAddress() {
        std::cout << "User Address: " << street_ << ", " << city_ << ", " << zipCode_ << std::endl;
    }

private:
    std::string name_;
    std::string street_;  // 🟡 数据泥团成员
    std::string city_;    // 🟡
    std::string zipCode_; // 🟡
};

// 订单配送类（重复地址字段）
class Delivery {
public:
    Delivery(const std::string& itemName, 
             const std::string& street, 
             const std::string& city, 
             const std::string& zipCode)
        : itemName_(itemName), street_(street), city_(city), zipCode_(zipCode) {}

    void ship() {
        std::cout << "Shipping " << itemName_ << " to: " 
                  << street_ << ", " << city_ << ", " << zipCode_ << std::endl;
    }

private:
    std::string itemName_;
    std::string street_;  // 🟡 重复的数据泥团
    std::string city_;    // 🟡
    std::string zipCode_; // 🟡
};

// 客户端代码
int main() {
    User user("Alice", "123 Main St", "New York", "10001");
    user.printAddress(); // 输出地址信息

    Delivery delivery("Laptop", "123 Main St", "New York", "10001");
    delivery.ship(); // 输出配送信息

    return 0;
}
```

---

### 问题分析
1. **数据泥团表现**：
    - `street`、`city`、`zipCode` 这三个字段总是同时出现在 `User` 和 `Delivery` 类中。
    - 地址信息的处理逻辑（如打印、拼接）在多个类中重复。
2. **维护成本高**：
    - 若地址格式需要修改（例如增加国家字段），需在所有使用这些字段的类中逐个修改。
3. **缺乏业务抽象**：
    - 地址信息没有被抽象为一个独立的业务概念。

---

### 重构思路
1. **封装数据泥团**：
    - 将 `street`、`city`、`zipCode` 封装为 `Address` 类。
    - 将地址相关的行为（如格式化输出）移动到 `Address` 类中。
2. **替换重复字段**：
    - 在 `User` 和 `Delivery` 类中用 `Address` 对象替代原始字段。

---

### 重构后代码
```cpp
#include <iostream>
#include <string>

// 新封装的地址类
class Address {
public:
    Address(const std::string& street, 
            const std::string& city, 
            const std::string& zipCode)
        : street_(street), city_(city), zipCode_(zipCode) {}

    std::string format() const {
        return street_ + ", " + city_ + ", " + zipCode_;
    }

    // 未来可扩展方法（如验证邮编合法性）
    bool isValidZipCode() const {
        // 示例逻辑：假设邮编为5位数字
        return zipCode_.length() == 5 && 
               zipCode_.find_first_not_of("0123456789") == std::string::npos;
    }

private:
    std::string street_;
    std::string city_;
    std::string zipCode_;
};

// 重构后的用户类
class User {
public:
    User(const std::string& name, const Address& address)
        : name_(name), address_(address) {}

    void printAddress() {
        std::cout << "User Address: " << address_.format() << std::endl;
    }

private:
    std::string name_;
    Address address_; // 🟢 使用封装后的地址对象
};

// 重构后的配送类
class Delivery {
public:
    Delivery(const std::string& itemName, const Address& address)
        : itemName_(itemName), address_(address) {}

    void ship() {
        std::cout << "Shipping " << itemName_ << " to: " 
                  << address_.format() << std::endl;
    }

private:
    std::string itemName_;
    Address address_; // 🟢 复用地址对象
};

// 客户端代码
int main() {
    Address userAddress("123 Main St", "New York", "10001");
    User user("Alice", userAddress);
    user.printAddress(); 

    Delivery delivery("Laptop", userAddress);
    delivery.ship();

    // 验证邮编合法性（新增功能）
    std::cout << "Is valid zip code? " 
              << (userAddress.isValidZipCode() ? "Yes" : "No") << std::endl;

    return 0;
}
```

---

### 重构后优点
1. **消除数据泥团**：
    - 地址字段被封装在 `Address` 类中，`User` 和 `Delivery` 类不再重复定义这些字段。
2. **集中行为逻辑**：
    - 地址格式化、邮编验证等逻辑统一在 `Address` 类中实现。
    - 新增地址相关功能时，只需修改一个类。
3. **业务语义清晰**：
    - `Address` 类的存在明确表达了业务领域中的“地址”概念。
4. **可扩展性增强**：
    - 添加新字段（如国家）时，只需修改 `Address` 类，客户端代码不受影响。

---

### 输出结果
```plain
User Address: 123 Main St, New York, 10001
Shipping Laptop to: 123 Main St, New York, 10001
Is valid zip code? Yes
```

---

### 何时应该重构数据泥团？
1. **重复出现的字段组**：同一组字段在多个类/方法中出现。
2. **长参数列表**：多个方法需要传递相同的参数组合。
3. **隐藏的业务概念**：数据组对应现实世界中的实体（如地址、坐标、日期范围等）。

---

### 总结
数据泥团是典型的“**隐式概念缺失**”问题。通过重构：

1. **显式化抽象**：将隐式概念变为显式定义的类。
2. **提升内聚性**：相关数据和行为被集中管理。
3. **降低耦合度**：依赖方只需与封装后的对象交互，无需关心内部细节。

这种重构是领域驱动设计（DDD）中“**值对象**”（Value Object）概念的典型应用，能够显著提高代码的表达力和可维护性。
