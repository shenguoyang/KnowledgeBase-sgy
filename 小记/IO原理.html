🏁 存储基础概念

一、BIOS / UEFI vs MBR / GPT

一句话理清：BIOS、UEFI 是主板固件（开机最先运行的底层程序）， MBR、GPT 是硬盘分区表格式（硬盘怎么存分区信息）。 二者严格配对。

🔧 BIOS

老旧主板固件，16位实模式，纯文字界面，启动慢。只能搭配 MBR 分区表。

固件（管开机引导）

⚡ UEFI

现代主板固件，32/64位，图形界面，启动快，支持安全启动(Secure Boot)。优先搭配 GPT。

固件（管开机引导）

📋 MBR

老式分区表。上限 2TB 硬盘，最多 4个主分区。无CRC校验，无备份，损坏即丢失。

分区表（管硬盘分区）

📐 GPT

新式分区表。支持 超大硬盘(8ZB)，128个分区，自带首尾双备份+CRC32校验，更稳定。

分区表（管硬盘分区）

三种常用配对组合：

① BIOS + MBR

老旧电脑通用。Win7及更早系统标配。不能装Win11。

传统标配

② UEFI + GPT

现在主流标配。Win10/Win11、新电脑、固态硬盘都用这个。

现代标配 ✓

③ UEFI(CSM) + MBR

UEFI主板开启CSM兼容模式模拟旧BIOS。兼容老系统，不推荐Win11。

兼容模式
启动模式分区格式是否常用适用系统
BIOSMBR✅ 老旧标配Win7及更早
BIOSGPT❌ 不支持无法启动
UEFIMBR⚠️ 兼容模式老系统兼容
UEFIGPT✅ 现代标配Win10/11、新电脑
装机口诀：装Win11 → 必须 UEFI + GPT；装Win7老机 → BIOS + MBR；固态硬盘新机 → 一律 UEFI + GPT。 硬性规则：BIOS不认GPT硬盘，Win11强制要求 UEFI+GPT+安全启动。
生活比喻：BIOS/UEFI 是大门门锁系统（管怎么进门），MBR/GPT 是房间划分规则（管进门后怎么分配空间）。 老门锁只能配老房间划分，新门锁最优配新标准房间划分。

二、MBR 分区类型：主分区 · 扩展分区 · 逻辑分区

仅 MBR 分区表才有这三类划分，GPT 没有扩展/逻辑分区概念，直接无限建主分区。

🏠 主分区 (Primary)

可以直接装系统、开机引导的分区，电脑能单独识别启动。
MBR最多4个主分区。

类比：独立院子，能住人当家

🚪 扩展分区 (Extended)

MBR主分区不够用时，划出一块"中转区域"。本身不能存文件、不能用，只能用来拆分更小分区。
一个磁盘最多1个扩展分区。

类比：走廊空地，本身没法住人

🚪🚶 逻辑分区 (Logical)

在扩展分区里面拆分出来的小分区。日常存文件、软件都用它。无法单独引导系统。
数量无硬性限制。

类比：走廊里隔出的小房间
关键规则：MBR 超过4个分区需求时 → 必须先建扩展分区 → 再在扩展分区内拆分逻辑分区。 GPT 直接用主分区即可，不存在扩展/逻辑分区概念。新电脑 GPT 格式不用纠结。

三、Windows 四种动态卷

把多块物理硬盘灵活组合使用。基础磁盘只有简单卷，动态磁盘才有后三种卷。

📦 简单卷 (Simple)

最普通、最常用。单块硬盘上划分空间，一块盘可以有多个简单卷。删改不影响其他硬盘。日常C盘D盘都是简单卷。

稳定 · 日常标配

🔗 跨区卷 (Spanned)

多块硬盘空余空间拼接成一个盘符。容量叠加但无提速、无备份。任意一块硬盘损坏，全部数据丢失。

容量拼合 · 安全性低

⚡ 带区卷 (RAID 0)

数据拆分分散存到多块硬盘，多盘同时读写。速度大幅翻倍，游戏、剪辑常用。无备份，一块坏全废。

速度最快 · 零安全

🪞 镜像卷 (RAID 1)

两块硬盘实时一模一样复制。100%数据备份，一块坏掉另一块完好。可用容量 = 单块硬盘大小。

最高安全 · 容量减半
类型占用硬盘数速度数据安全主要用途
简单卷单块盘正常稳定个人电脑日常分区
跨区卷多块盘一般极低整合零散空余空间
带区卷多块盘最快极低追求高速读写
镜像卷两块盘普通最高重要数据备份防损坏
实用建议：家用普通电脑 → 只用简单卷，稳定够用；硬盘空间零散 → 少量用跨区卷；视频剪辑/大型游戏 → 带区卷； 公司重要文件/珍贵资料 → 镜像卷保障数据不丢失。

四、分区 vs 卷 —— 物理边界与逻辑接口

📐 分区 (Partition) — 物理层概念

在磁盘上划出的一段连续物理扇区边界，定义在分区表（MBR/GPT）里。由磁盘驱动 (Disk.sys) 直接管理，只懂扇区。

📦 卷 (Volume) — 逻辑层对象

对分区（或多个分区）的进一步封装，是文件系统能直接对话的标准化接口。由卷管理器提供，文件系统只认卷、只认LCN。

生活比喻：硬盘是一块地，分区就像用围墙隔出的物理地块。 卷则是在这些地块上铺设好的逻辑路面，文件系统可以在上面走路。

简单卷：一个地块上只铺一层路 → 看起来是一起的。 跨区卷：路直接跨过围墙，把两块完全不相邻的地块，铺成一条平整连贯的大道。

驱动栈分层模型

💿 底层：磁盘驱动 Disk.sys

直接和硬件对话，只懂扇区。读取MBR/GPT分区表，为每个分区创建独立的物理设备对象（\Device\Harddisk0\Partition1）。

🗂️ 中层：卷管理器 Volume Manager

卷能跨区的关键拼接层。简单卷：1:1映射不做转换。跨区卷：拼接多个分区为统一卷设备对象，负责将卷的LCN映射到对应物理分区的LBA。

📁 顶层：文件系统 NTFS.sys

只认卷，不管底层是单分区还是跨区。所有操作只认逻辑簇号(LCN)。最终把"读卷的第X个簇"发给卷管理器即可，扇区物理寻址完全在视野之外。

核心认知：文件系统做的核心转换是 文件名 + 偏移 → 卷逻辑簇 (LCN)。 至于这个 LCN 最终对应哪块盘、哪个扇区，文件系统完全不管——那是卷管理器和磁盘驱动的事。

驱动栈分层说明

💿 磁盘驱动 (Disk.sys)

直接和硬件对话，只懂扇区。读取 MBR/GPT 分区表，解析出存有起止LBA的 PARTITION_ENTRY 结构。为每个识别出的分区创建独立物理设备对象。

🗂️ 卷管理器 (VolMgr)

卷能跨区的关键拼接层。简单卷：1:1映射，簇到扇区 = 物理起始LBA + 簇偏移。跨区卷：等所有成员分区就绪后拼接成统一卷设备，当读写卷的逻辑簇时算出在哪个物理盘的分区上。

📁 文件系统 (NTFS.sys)

只认卷，完全不用管底层是单分区还是跨区。直接挂载在卷管理器提供的逻辑卷设备对象上。所有操作只认逻辑簇号(LCN)，向$MFT记录占用LCN，向$Bitmap查询空闲LCN。

代码分层视角：
① 文件系统：文件名 + 偏移 → VCN → LCN（查 MFT → DataRuns）
② 卷管理器：LCN → LBA（分区偏移计算 / 动态卷布局变换）
③ 磁盘驱动：LBA → 物理扇区读写（构造 SRB / DMA）

📁 NTFS 文件系统深度解析

NTFS 详解

📐 代码级映射链：从文件名到物理扇区

文件系统层核心做的就是 "文件名 + 偏移" → "卷逻辑簇" 的转换：
① FileName → MFT Record  （OpenFile → 目录B+树查找 → MFT记录号）
② FileOffset → VCN  （偏移 / 簇大小 = 虚拟簇号）
③ VCN → LCN  （查MFT记录的DataRuns/RunList → 遍历找对应Run）
④ LCN → 卷管理器 → 物理扇区  （卷管理器计算分区偏移+簇偏移 → LBA） 至于这个 LCN 最终对应哪块盘、哪个扇区，文件系统完全不管——那是卷管理器和磁盘驱动的事。
