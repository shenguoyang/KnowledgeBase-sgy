🎬 底层原理可视化演示

以下演示帮助理解内存映射如何消除磁盘碎片，以及为什么 memcpy 比 WriteFile 更快。

📌 第一幕：碎片是怎么产生的？

想象磁盘是一排储物柜，每来一辆车就创建一个新文件来存图像。随着文件不断创建和删除，磁盘上出现大量"空洞"——这就是磁盘碎片。

⬇ 传统方式：每张图一个文件，随意创建/删除

正常文件 空闲空间 碎片空洞
关键洞察：每张图一个小文件 → 日积月累 → 磁盘变成"瑞士奶酪" → 新文件找不到整块空间 → 被"撕碎"存放 → 磁头疯狂跳动 → 写入速度越来越慢。

📌 第二幕：预分配——先占坑、再填坑

本方案的核心思想：启动时一次性预分配整个大文件，然后通过内存映射把它变成一块"虚拟内存"。写入新图像只需 memcpy，写满后自动回到开头覆盖最早的数据。

⬇ 环形缓冲区 + 内存映射：通过指针直接读写，零系统调用

空闲 Block 已写入 被覆盖 🔴 写入指针

📌 第三幕：文件里面长什么样？

整个文件采用两区布局，元数据集中在文件前部，与 Block 数据区分开。就像一本带目录的相册：

Header（文件头 — 4KB） BlockMeta 区（元数据集中存放） Block 区（变长图像数据）
为什么 BlockMeta 集中在前面？崩溃恢复时顺序读取 BlockMeta 区即可获得全部元数据，无需跳跃扫描。写入时 BlockMeta 紧邻 Header，一次 Seek 即可更新两者。

📌 第四幕：memcpy 为什么比 WriteFile 快？

传统 WriteFile 需要用户态 → 内核态切换，数据先拷到内核缓冲区再写磁盘。内存映射直接把文件页映射到用户地址空间，一次 memcpy 即完成写入。

WriteFile 路径（2 次拷贝 + 内核切换） memcpy 路径（0 次额外拷贝，0 次内核切换）
零拷贝（Zero-Copy）的含义：不是说不拷贝数据，而是省去了内核中转。以下逐步骤对比：
步骤WriteFile 方案（方案一/二）内存映射方案（方案三）
① 定位 SetFilePointerEx(hFile, offset)
→ 用户态→内核态→返回		 ptr = mappedView + offset
→ 纯用户态指针运算，0 次系统调用		 ✅ 跳过
② 写入 WriteFile(hFile, buf, size)
→ 用户态→内核态→I/O管理器→拷到内核缓冲区→返回		 memcpy(mappedView+offset, buf, size)
→ CPU 直接写入文件缓存页，硬件 MMU 完成地址翻译		 ✅ 跳过
③ 中间拷贝 用户缓冲区 → 内核缓冲区 → 文件缓存
→ 至少 1 次额外内存拷贝		 用户缓冲区 → 文件缓存页
→ 直接拷贝到目标，无中转		 ✅ 跳过
④ 刷盘 FlushFileBuffers(hFile)
→ 用户态→内核态，强制回写		 OS Lazy Writer 自动异步刷盘
→ 无需调用（关键节点可选 FlushViewOfFile）		 ✅ 通常跳过

一次写入从 2 次 Seek + 2~3 次 WriteFile + FlushFileBuffers ≈ 5~6 次系统调用，降至 1 次 memcpy ≈ 0 次系统调用。用户态↔内核态切换从约 10~12 次降为 0。

📌 第五幕：虚拟内存 ≠ 物理内存

很多人担心：「文件 2TB，内存映射岂不是要 2TB 内存？」——这是对内存映射最常见的误解。

❌ 错误理解

MapViewOfFile(2TB) → 消耗 2TB 物理 RAM → ❌ 不可能！

✅ 正确理解

MapViewOfFile(2TB) → 消耗 2TB 虚拟地址空间（64 位有 128TB，够用）
物理内存仅在实际访问时按需分配（4KB 页粒度）
写入时只需当前操作的那几页在物理内存中（几 KB~几 MB）
任务管理器显示「虚拟内存」飙升到 40GB？不必担心。那是虚拟地址空间占用，不是物理 RAM。真正需要关注的指标是「工作集」（物理内存占用）和「提交大小」——这两项几乎不受文件映射影响。详见下文内存映射原理章节的完整解释。
已映射的虚拟页（在物理内存中） 已映射的虚拟页（未访问，不在物理内存） 脏页（已修改，待 OS 刷回磁盘）

启动时磁盘整理与空间独占

为什么需要启动整理？

单纯预分配文件，NTFS 会尽力分配连续空间，但如果磁盘已经碎片化，NTFS 可能只能给碎片化的空间。因此需要在程序启动时主动分析、整理、验证，确保获得物理连续的磁盘区域。

五步启动序列

┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ ① 分析   │───▶│ ② 创建+预分配│───▶│ ③ 验证连续性 │───▶│ ④ 碎片整理  │───▶│ ⑤ 独占锁定  │
│ 空闲空间 │    │   环形文件    │    │  (extent数)  │    │  (如需)     │    │  (防其他程序)│
└──────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘

① 分析空闲空间 — FSCTL_GET_VOLUME_BITMAP

向卷发送 IOCTL 获取空闲簇位图，每一位代表一个簇的分配状态。遍历位图找到最大的连续空闲区段，判断是否 ≥ 目标文件大小。

② 创建文件并预分配 — SetFileValidData

用 CreateFile(dwShareMode=0) 独占创建文件，然后 SetFileValidData 瞬时标记文件有效数据大小（需 SE_MANAGE_VOLUME_NAME 特权）。NTFS 立刻为全部字节分配物理簇。无特权时回退到 SetEndOfFile + 分块写零（64MB Overlapped I/O）。

③ 验证物理连续性 — FSCTL_GET_RETRIEVAL_POINTERS

查询文件的物理 extent 布局：

Extent 数量评级处理
= 1完美连续无需处理，直接进入运行
2 ~ 3可接受记录日志，继续运行
≥ 4需要整理触发 FSCTL_MOVE_FILE 强制重排

④ 碎片整理 — FSCTL_MOVE_FILE

如果 extent 数量超标，向卷发送 FSCTL_MOVE_FILE，让 NTFS 将文件簇重新布局到连续区域。如果仍失败 → 删除文件重建。

⑤ 独占锁定 — dwShareMode=0 + LockFileEx

层级机制效果
文件系统层NTFS 簇分配其他文件无法占用此区域（簇已分配给我们的文件）
文件句柄层dwShareMode=0其他进程无法打开/读取/写入/删除此文件
字节锁层LockFileEx进程内对整个文件加排他锁
关键安全保障：dwShareMode=0 意味着即使资源管理器也无法读取此文件。文件已预分配，其占用的物理扇区被 NTFS 标记为已分配，其他程序根本无法写到这里。这构成了真正的空间隔离。

背景与目标

场景描述

Windows 环境下车辆缺陷检测应用，每辆车到达时需存储检测图像。图像大小在 1MB~3MB 之间波动。磁盘大部分空间用于存储图像，配置数据极少。

核心问题

长时间运行后磁盘碎片严重 → 写入速度下降 → 影响检测效率

本方案特点

在预分配单文件基础上，利用 Windows Memory-Mapped File 机制将文件映射到进程虚拟地址空间，以 memcpy 替代 WriteFile，实现零系统调用的写入路径。

目标

✅通过预分配消除磁盘碎片
✅环形覆盖策略，磁盘空间占用有界
✅memcpy 写入，零内核切换，性能极致
✅支持变长存储，空间利用率 ~99%
✅封装为独立 C++ 组件库

总体架构

核心思路：将整个图像存储抽象为一个内存映射环形缓冲区（Memory-Mapped Ring Buffer）。程序启动时在 NTFS 上一次性预分配一个大文件，通过 CreateFileMapping + MapViewOfFile 映射到进程虚拟地址空间。之后所有读写均通过指针操作，无需任何文件 I/O API。

┌──────────────────────────────────────────────────────────────┐
│                  预分配文件（单一文件，物理连续）                │
├────────┬──────────────────────┬──────────┬─────┬──────────────┤
│ Header │ BlockMeta[0..N-1]    │ Block 0  │ ... │ Block N-1    │
│ (4KB)  │ (N × sizeof(Meta))  │ (变长)    │     │ (变长)        │
└────────┴──────────────────────┴──────────┴─────┴──────────────┘
▲        ▲                      ▲
│        │                      │
│        │                      └─ Block 区：实际图像数据
│        └─ BlockMeta 区：元数据集中存放（时间戳、大小、校验值）
└─ Header：魔数/版本/写指针/块大小等控制信息

映射后：mappedView 指针指向文件开头
  mappedView[0..4095]              → Header
  mappedView[4096..4096+metaSize]  → BlockMeta 数组
  mappedView[4096+metaSize..]      → Block 数据区
区域大小说明
Header4KB文件头，存储魔数/版本/写指针/块大小/块数量等控制信息
BlockMeta 区N × 64B每个 Block 的元数据（时间戳、实际大小、校验值），集中存放于文件前部
Block 区N × BlockSize每个 Block 最大 BlockSize 字节，实际存储 ≤ BlockSize 的图像数据（变长）
设计考量：BlockMeta 集中在 Header 之后，而非分散在每个 Block 后面。这样做的好处：① 崩溃恢复时顺序读取即可获得全部元数据；② BlockMeta + Header 在同一次缓存行回写中更新，减少磁盘 I/O。

块大小选择

推荐：BlockSize = 最大图像尺寸 × 1.2

BlockSize = 3MB × 1.2 ≈ 3.6MB → 向上取整到 4MB

每张图像占用 ≤ BlockSize 的可变空间（变长存储）。优点：空间利用率 ~99%，实现简单。极端情况图像超过 BlockSize 需拆分存储（罕见）。

内存映射原理

核心概念：虚拟内存 ≠ 物理内存

这是对内存映射最常见的误解。以下用具体数字澄清：

虚拟地址空间物理内存 (RAM)
MapViewOfFile 映射 2TB 文件消耗 2TB（64 位有 128TB）仅按需分配（4KB 页粒度）
写入一张 2MB 图像时不变（已映射）仅 2MB 对应页面在 RAM 中
文件其余 99.9% 区域已映射但未访问0 字节物理内存占用

任务管理器的"虚拟内存"飙升：正常且无害

映射 40GB 文件后，Windows 任务管理器中该进程的「虚拟内存」列会显示 +40GB。这是虚拟地址空间占用，不是物理内存消耗。

任务管理器指标含义映射 40GB 后是否值得担心
虚拟大小 (Virtual Size) 进程已保留的虚拟地址空间总量 +40GB 不用担心 — 64 位有 128TB 空间
物理内存 / 工作集 (Working Set) 实际占用的物理 RAM +几十 MB（仅当前访问的页） 不用担心
提交大小 (Commit Size) 需要页面文件或物理内存保证的私用内存 几乎不变 不用担心 — 文件映射不消耗 Commit
关键区别：MapViewOfFile 映射的页面由磁盘文件本身作为后备存储（Backing Store），不走页面文件（pagefile.sys）。OS 随时可以从文件读回页面内容，不需要 Commit 保证。这就是文件映射和 malloc(40GB) 的本质区别—— 后者会实实在在消耗 40GB 的提交内存（需要物理 RAM + 页面文件来保证），而前者几乎不消耗提交内存。

五层映射架构

① 用户态虚拟地址空间
进程看到的是一块连续内存（如 0x1E00000000），通过指针直接读写
▼ 页表映射 (MMU)
② 页表（Page Table）
CPU 将虚拟地址翻译为物理地址。未访问的页标记为「缺页」
▼ 内存管理器
③ 内存管理器（Memory Manager）
Windows 内核管理 Section Object，将文件页面映射到物理内存
▼ 缓存管理器
④ 缓存管理器（Cache Manager）
Lazy Writer 线程每秒将脏页异步刷回磁盘
▼ 后备存储
⑤ 磁盘文件（Backing Store）
NTFS 上的实际文件，物理连续的单文件

缺页中断（Page Fault）

首次访问某个映射页时，CPU 触发缺页中断 → OS 从磁盘读取该页到物理内存 → 建立页表映射 → 进程继续执行（对进程透明）。这个过程约 1~5ms，但每个页只发生一次。

预触发达（Prefaulting）：如果对首次写入延迟极度敏感，可以在映射后立即 memset 遍历一遍映射区域，强制触发全部缺页中断。但通常不必要——写入路径上的缺页延迟远小于传统 WriteFile 的内核切换开销。

Lazy Writer 异步刷盘 + 进程崩溃安全性

修改映射内存后，对应页被标记为"脏页"，由 Windows 内核态的 Lazy Writer 线程管理。脏页位于系统文件缓存（File System Cache）中，属于内核资源，不随进程退出而丢失。

进程崩溃 ≠ 数据丢失：应用程序 crash 后，进程的虚拟地址空间被销毁，但 Lazy Writer 仍会将脏页刷回磁盘。只有操作系统崩溃或断电才会丢失尚未刷盘的脏页。这与 WriteFile + FlushFileBuffers 面临的物理限制相同。如需更强的持久性保证，可在每次写入后调用 FlushViewOfFile 强制立即刷盘。

关键算法

初始化（首次启动 / 文件不存在）

Init(file_path, block_count, block_size):
  1. metaSize = block_count × sizeof(BlockMeta)
     totalSize = 4096 + metaSize + block_count × block_size
  2. 创建文件并预分配（CreateFile + SetFileValidData / SetEndOfFile）
  3. 创建文件映射对象:
     hMapping = CreateFileMapping(hFile, NULL, PAGE_READWRITE, sizeHi, sizeLo, NULL)
  4. 映射到虚拟地址空间:
     mappedView = (uint8_t*)MapViewOfFile(hMapping, FILE_MAP_READ | FILE_MAP_WRITE, 0, 0, 0)
     // ↑ 此后所有操作均通过 mappedView 指针！
  5. 初始化 Header: magic / version / block_size / block_count
  6. BlockMeta 区清零
  7. FlushViewOfFile(mappedView, totalSize)

写入图像（核心路径）

WriteImage(data, size, vehicle_id):
  前提: size ≤ block_size

  1. 获取写锁 (SRWLock Exclusive)
  2. 通过指针计算偏移（无需 Seek！）:
     metaOffset  = 4096 + writeCursor × sizeof(BlockMeta)
     blockOffset = 4096 + metaSize + writeCursor × blockSize
  3. 写入图像数据:
     memcpy(mappedView + blockOffset, data, size);        // ← 就这一行！
  4. 写入 BlockMeta:
     BlockMeta* meta = (BlockMeta*)(mappedView + metaOffset);
     meta->timestamp   = GetSystemTimeAsFileTime();
     meta->actual_size = size;
     meta->flags       = 0x01;
     wcscpy_s(meta->vehicle_id, 16, vehicle_id);
     meta->data_crc32  = crc32(data, size);
  5. 更新 Header:
     header->write_cursor = (writeCursor + 1) % blockCount;
     header->total_blocks_written++;
  6. 如需确认落盘: FlushViewOfFile(mappedView + blockOffset, size)
     // 通常不需要——OS 的 Lazy Writer 会自动刷盘
  7. 释放写锁

读取图像

ReadByIndex(blockIndex):
  1. 获取读锁 (SRWLock Shared)
  2. BlockMeta* meta = (BlockMeta*)(mappedView + 4096) + blockIndex;
  3. if (meta->flags != 0x01) return 空;
  4. uint64_t blockOffset = 4096 + metaSize + blockIndex × block_size;
  5. 校验 CRC32(mappedView + blockOffset, meta->actual_size) == meta->data_crc32
  6. 拷贝返回: memcpy(outBuffer, mappedView + blockOffset, meta->actual_size);
  7. 释放读锁

按车辆 ID 检索

ReadByVehicle(vehicle_id):
  1. 获取读锁 (SRWLock Shared)
  2. BlockMeta* metas = (BlockMeta*)(mappedView + 4096);
  3. 遍历 metas[0..N-1]，匹配 vehicle_id → 记录匹配的 blockIndex
  4. 对每个匹配项调用 ReadByIndex 读取图像数据
  5. 释放读锁

环形覆盖语义

当 write_cursor 绕回时，该位置原有的 BlockMeta.flags 非零说明有旧数据，直接覆盖即可——无需显式删除。通过 header->total_blocks_written 判断数据新旧：数字越大越新。

UI 显示线程读取最新图像按需（中频）共享读锁 (SRWLock Shared)
分析/回放线程按车辆 ID 读取按需（低频）共享读锁 (SRWLock Shared)
注意：memcpy 不保证原子性。虽然写入路径没有系统调用，但多个线程同时 memcpy 到不同 Block 仍可能产生竞争（Header 的 write_cursor 更新不是原子的）。仍需用 SRWLock 保护写操作。读操作可以并发（Shared Lock）。

异常恢复

内存映射的崩溃优势：脏页由 OS 内核管理，进程崩溃后 Lazy Writer 仍会将已 memcpy 的数据刷回磁盘（脏页在系统文件缓存中，不随进程销毁）。只有断电或 OS 崩溃才会丢失未刷盘数据——即便 WriteFile 方案也面临同样限制。实际恢复时，只需扫描 BlockMeta 区即可重建完整状态。
崩溃时机后果恢复方式
memcpy 途中当前 Block 数据不完整CRC 校验失败 → 标记该 Block 为 Empty，下个写入覆盖
写完数据、写 BlockMeta 前数据完整但无元数据扫描 BlockMeta 发现 flags=Empty，该 Block 可被覆盖
写完 BlockMeta、更新 Header 前write_cursor 未更新扫描 BlockMeta 找到最后 Valid 块+1，重建 write_cursor

恢复流程（每次启动执行）

Recover(file_path):
  1. 打开文件，创建映射
  2. 验证 Header magic 和 CRC32
  3. 遍历 BlockMeta 区所有条目:
     → 找到最后一个 flags=Valid 的条目 → 重建 write_cursor
  4. 对所有声称 Valid 的条目校验 data_crc32:
     → CRC 不匹配 → 标记 flags=Empty, actual_size=0
  5. 写回修正后的 Header
  6. FlushViewOfFile
恢复性能优势：BlockMeta 集中存放于文件前部，恢复时连续顺序读取即可获得全部元数据，无需像方案一那样在 Block 间隙中跳跃扫描。对于 10,000 个 Block，扫描仅需读取 640KB 连续内存。

// 2. 映射到进程地址空间 — 整个文件！
uint8_t* view = (uint8_t*)MapViewOfFile(hMapping, FILE_MAP_READ | FILE_MAP_WRITE, 0, 0, 0);

// 3. 之后所有操作都通过 view 指针，无需任何文件 API

// 4. 可选：强制刷盘

FlushFileBuffers(hFile);                  // 刷整个文件

分段映射策略（仅 32 位需要）

const uint64_t WINDOW_SIZE = 256 * 1024 * 1024;  // 256MB 窗口
uint64_t blockOffset = HEADER_SIZE + metaSize + writeCursor * blockSize;
uint64_t windowBase = (blockOffset / WINDOW_SIZE) * WINDOW_SIZE;

UnmapViewOfFile(mappedView);
mappedView = (uint8_t*)MapViewOfFile(hMapping,
    FILE_MAP_READ | FILE_MAP_WRITE,
    (DWORD)(windowBase >> 32), (DWORD)(windowBase & 0xFFFFFFFF), WINDOW_SIZE);
// 此后 mappedView[0] 对应文件的 windowBase 偏移
// 写入: memcpy(mappedView + (blockOffset - windowBase), data, size);

优缺点与对比

独特优点

⚡ 零系统调用写入memcpy 替代 WriteFile，无用户态↔内核态切换，性能极致
⚡ 极简代码写入仅需一行 memcpy + Header/BlockMeta 更新（指针操作）
⚡ OS 自动回写Lazy Writer 线程异步刷脏页，无需手动 Flush（关键节点可选）
⚡ 读写统一模型读/写均通过指针，无需 Seek，无需 ReadFile/WriteFile
⚡ 变长存储每 Block 可存储 ≤ blockSize 的任意大小，空间利用率 ~99%
⚡ BlockMeta 集中崩溃恢复顺序读取文件前部即可，无需跳跃扫描

独特缺点

⚠ 32 位受限虚拟地址空间 2~4GB，需分段映射（增加复杂度）
⚠ 缺页中断延迟首次访问新页触发 Page Fault（可预触发达避免）
⚠ 刷盘时机不确定OS 决定何时回写，断电可能丢失未刷数据（可调 FlushViewOfFile 频率）
⚠ 不兼容 FILE_FLAG_NO_BUFFERING内存映射与无缓冲 I/O 互斥，无法绕过系统缓存

// MMapRingBufferTypes.h
#pragma once
#include <cstdint>

struct MMapRingBufferConfig {
    wchar_t  drive_letter;           // 磁盘盘符，如 L'D'
    uint32_t block_size;             // 每个 Block 最大字节数 (例: 4MB = 4194304)
    uint32_t block_count;            // Block 总数量 (例: 10000)
    uint32_t max_extents_accepted;   // 可接受的最大 extent 数量 (默认 3)
    bool     use_mapping;            // true = 内存映射, false = WriteFile 回退
};

struct ImageInfo {
    uint64_t timestamp;         // FILETIME 写入时间
    uint32_t block_index;       // Block 索引
    uint32_t actual_size;       // 图像实际字节数
    wchar_t  vehicle_id[16];    // 车辆标识
    uint8_t* data;              // 图像数据（组件分配，调用方 FreeImage 释放）
};

struct RingBufferStats {
    uint32_t block_size, block_count;
    uint32_t used_blocks, free_blocks;
    uint32_t write_cursor;
    uint64_t total_blocks_written, total_bytes_written;
    uint64_t total_file_size, mapped_size;
    bool     is_mapped;            // 是否使用内存映射
    bool     is_healthy;
    uint32_t extent_count;         // 文件物理 extent 数量
    bool     is_contiguous;        // 是否完全连续
    uint64_t largest_free_extent;
};

MMapRingBuffer 接口

// MMapRingBuffer.h
#pragma once
#include "MMapRingBufferTypes.h"
#include <memory>

class MMapRingBuffer {
public:
    // ═══ 生命周期 ═══
    MMapRingBuffer();
    ~MMapRingBuffer();                          // 自动调用 Shutdown()
    // 禁止拷贝和移动
    MMapRingBuffer(const MMapRingBuffer&) = delete;
    MMapRingBuffer& operator=(const MMapRingBuffer&) = delete;
    MMapRingBuffer(MMapRingBuffer&&) = delete;
    MMapRingBuffer& operator=(MMapRingBuffer&&) = delete;

    // ═══ 初始化与关闭 ═══
    /// @brief 初始化环形缓冲区组件
    /// @param cfg  配置参数（盘符 / block大小 / block个数 / 连续性阈值）
    /// @return MMRBF_OK 成功，其他为错误码
    /// @note  组件在指定盘符根目录自动创建数据文件
    ///        启动流程：分析空闲空间 → 预分配 → 验证物理连续性 → 内存映射 → 独占锁定
    ///        映射成功后，所有读写直接通过指针，无需任何文件 API
    ///        HDD 典型耗时：40GB 预分配约 2~3 分钟，映射瞬时完成
    uint32_t Init(const MMapRingBufferConfig& cfg);

    /// @brief 关闭环形缓冲区，解除映射，释放文件锁，确保数据落盘
    /// @note  析构函数自动调用。关闭后可再次 Init 重新初始化
    void Shutdown();

    // ═══ 写入 ═══
    /// @brief 写入一张图像（内部仅一次 memcpy + 元数据更新）
    /// @param data       图像数据指针
    /// @param size       图像字节数 (≤ block_size)
    /// @param vehicle_id 车辆标识（可选，最多 15 个宽字符）
    /// @param out_index  输出：写入的 Block 索引（可选）
    /// @return MMRBF_OK 成功
    uint32_t Write(const void*    data,
                   uint32_t       size,
                   const wchar_t* vehicle_id = nullptr,
                   uint32_t*      out_index = nullptr);

    // ═══ 读取 ═══
    /// @brief 按 Block 索引读取图像
    uint32_t ReadByIndex(uint32_t  block_index,
                         void*     buffer,
                         uint32_t  buffer_size,
                         uint32_t* out_actual_size);

    /// @brief 按车辆 ID 检索图像（返回匹配数量）
    int32_t  ReadByVehicle(const wchar_t* vehicle_id,
                           ImageInfo*     results,
                           int32_t        max_results);

    /// @brief 读取最近 N 张图像
    int32_t  ReadLatest(int32_t    count,
                        ImageInfo* results,
                        int32_t    max_results);

    // ═══ 统计与工具 ═══
    void     GetStats(RingBufferStats* out_stats) const;
    static void      FreeImage(ImageInfo* image);

    /// @brief 强制刷盘（调用 FlushViewOfFile）
    void     Flush();

    /// @brief 验证文件完整性（Header CRC + BlockMeta CRC 扫描）
    static bool      ValidateFile(const wchar_t* file_path);

    /// @brief 计算预分配文件总大小
    static uint64_t  CalculateFileSize(uint32_t block_size, uint32_t block_count);

    /// @brief 获取映射指针（高级用户直接操作，慎用）
    /// @note  返回 mappedView 指针，调用方可直接 memcpy 读写
    ///        使用前必须持有锁（GetWriteLock / GetReadLock）
    uint8_t* GetMappedView();

private:
    class Impl;
    std::unique_ptr<Impl> impl_;
};

使用示例

// ═══ 应用启动 ═══
MMapRingBuffer storage;
MMapRingBufferConfig cfg = {};
cfg.drive_letter         = L'D';
cfg.block_size           = 4 * 1024 * 1024;   // 4MB per block
cfg.block_count          = 10000;             // 10000 blocks ≈ 40GB
cfg.max_extents_accepted = 3;
cfg.use_mapping          = true;              // 启用内存映射

uint32_t err = storage.Init(cfg);
if (err != MMRBF_OK) {
    // MMRBF_ERR_TOO_FRAGMENTED → HDD 碎片严重，需手动整理
    // MMRBF_ERR_ALREADY_LOCKED  → 已有实例在运行
    // MMRBF_ERR_MAP_FAILED      → 映射失败（32位超大文件等）
    return err;
}

// ═══ 采集线程：来车写入 ═══
void OnVehicleArrive(const uint8_t* imageData, uint32_t size, const wchar_t* plate) {
    uint32_t idx = 0;
    uint32_t err = storage.Write(imageData, size, plate, &idx);
    // ↑ 内部仅一次 memcpy + 元数据更新，0 次内核切换
    if (err == MMRBF_OK) {
        // 写入成功，idx 为 Block 索引
    }
}

// ═══ UI 线程：查看最新图像 ═══
ImageInfo latest[10];
int count = storage.ReadLatest(10, latest, 10);
for (int i = 0; i < count; i++) {
    DisplayImage(latest[i].data, latest[i].actual_size);
    storage.FreeImage(&latest[i]);  // 释放组件分配的内存
}

// ═══ 检索线程：按车牌查询 ═══
ImageInfo results[100];
int n = storage.ReadByVehicle(L"京A12345", results, 100);
// 遍历 results，使用后逐个 FreeImage

// ═══ 应用退出 ═══
storage.Shutdown();  // 解除映射 + 刷盘 + 释放锁
// 或让析构函数自动处理
