🎬 底层原理可视化演示

以下演示帮助不懂磁盘碎片和环形缓冲区的新人理解本方案的核心思路。

📌 第一幕：碎片是怎么产生的？

想象磁盘是一排连续的储物柜，每个柜子大小固定（比如 4KB）。当你不断存入大小不一、时间不一的数据，又不时删除一些时，磁盘上就出现了很多"空洞"——这就是磁盘碎片。

⬇ 传统方式：每张图一个文件，随意创建/删除

正常文件 空闲空间 碎片空洞（难以利用）
关键洞察：每张图一个小文件 → 日积月累 → 磁盘变成"瑞士奶酪" → 新文件找不到整块空间 → 被"撕碎"存放 → 磁头疯狂跳动 → 写入速度越来越慢。这就是你要解决的根本问题。

📌 第二幕：预分配——先占坑、再填坑

本方案的核心思想很像电影院：开场前就把整个放映厅的座位全包下来（预分配），来一个观众坐一个位置（写入），坐满了怎么办？回到第一排重新开始（环形覆盖）。

⬇ 环形缓冲区：固定大小，写满后回到开头继续

空闲 Block 已写入 (序号 N) 被覆盖 (旧数据被新数据替换) 🔴 写入指针 🟢 最新数据

📌 第三幕：文件里面长什么样？

整个文件由三部分组成。就像一个带目录的相册：

Header (目录页 — 4KB) Block 区 (照片页 — 每页一张图) Index 区 (索引页 — 记录每张照片的车牌和时间)
为什么 Index 区放在末尾？Block 区是热数据（每次都要写），放在一起减少磁头跳跃；Index 只在尾部追加更新。

📌 第五幕：空间写满了怎么办？——环形覆盖

这是本方案最精妙的地方。不需要"删除"操作——新的图像直接覆盖最旧的数据，就像时钟指针一样永远向前。

有效数据 本次被覆盖的旧数据 写入指针
sequence 的作用：文件中原先的数据可能有序列号 1~10000，写满一圈后，第 1 块被序列号 10001 覆盖。 判断数据新旧不看物理位置，看 sequence 数字大小即可。

📌 第六幕：启动时磁盘整理 + 空间独占

应用启动时，首先分析磁盘空闲空间布局，找到最大的连续空闲区域，然后预分配文件、验证连续性。如果碎片严重，自动触发文件簇重排（FSCTL_MOVE_FILE）。最后用 dwShareMode=0 独占锁定文件，其他程序无法读写。

启动时磁盘整理与空间独占

为什么需要启动整理？

单纯预分配文件，NTFS 会尽力分配连续空间，但如果磁盘已经碎片化，NTFS 可能只能给碎片化的空间。 因此需要在程序启动时主动分析、整理、验证，确保获得物理连续的磁盘区域。

五步启动序列

┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ ① 分析   │───▶│ ② 创建+预分配│───▶│ ③ 验证连续性 │───▶│ ④ 碎片整理  │───▶│ ⑤ 独占锁定  │
│ 空闲空间 │    │   环形文件    │    │  (extent数)  │    │  (如需)     │    │  (防其他程序)│
└──────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘

① 分析空闲空间 — FSCTL_GET_VOLUME_BITMAP

向卷发送 IOCTL 获取空闲簇位图，每一位代表一个簇的分配状态（0=空闲，1=已用）。遍历位图找到最大的连续空闲区段，判断是否 ≥ 目标文件大小。

② 创建文件并预分配 — SetFileValidData

用 CreateFile(dwShareMode=0) 独占创建文件，然后 SetFileValidData 瞬时标记文件有效数据大小（需 SE_MANAGE_VOLUME_NAME 特权）。NTFS 立刻为全部字节分配物理簇。无特权时回退到 SetEndOfFile + 分块写零。

③ 验证物理连续性 — FSCTL_GET_RETRIEVAL_POINTERS

查询文件的物理 extent 布局：

Extent 数量评级处理
= 1完美连续无需处理
2 ~ 3可接受记录日志
≥ 4需要整理触发步骤④

④ 碎片整理 — FSCTL_MOVE_FILE

如果 extent 数量超标，向卷发送 FSCTL_MOVE_FILE，让 NTFS 将文件簇重新布局到连续区域。如果仍失败 → 删除文件重建。

⑤ 独占锁定 — dwShareMode=0 + LockFileEx

层级机制效果
文件系统层NTFS 簇分配其他文件无法占用此区域（簇已分配给我们的文件）
文件句柄层dwShareMode=0其他进程无法打开/读取/写入/删除此文件
字节锁层LockFileEx进程内对整个文件加排他锁
关键安全保障：dwShareMode=0 意味着即使资源管理器也无法读取此文件。而且文件已预分配，其占用的物理扇区被 NTFS 标记为已分配，其他程序根本无法写到这里。这构成了真正的空间隔离。

背景与目标

场景描述

Windows 环境下车辆缺陷检测应用，每辆车到达时需存储检测图像。图像大小在 1MB~3MB 之间波动。磁盘大部分空间用于存储图像，配置数据极少。

核心问题

长时间运行后磁盘碎片严重 → 写入速度下降 → 影响检测效率

目标

✅通过预分配消除磁盘碎片
✅环形覆盖策略，磁盘空间占用有界
✅启动快速、写入高效、数据可靠
✅封装为独立 C++ 组件库，不改动原有应用程序

总体架构

核心思路：将整个图像存储抽象为一个文件级环形缓冲区（Ring Buffer File）。程序启动时在 NTFS 上一次性预分配一个大的连续文件，内部按固定块（Block）切分，写入指针循环移动，写满后自动覆盖最早的数据。

┌──────────────────────────────────────────────────────────────┐
│                      预分配文件 (单一文件)                     │
├────────┬──────────┬──────────┬─────┬──────────┬──────────────┤
│ Header │ Block 0  │ Block 1  │ ... │ Block N-1│   Index区    │
│ (4KB)  │ (4MB)    │ (4MB)    │     │ (4MB)    │ (N×128B)     │
└────────┴──────────┴──────────┴─────┴──────────┴──────────────┘
区域大小说明
Header4KB文件头，存储魔数/版本/写指针/序列号等控制信息
Block 区N × BlockSize每个 Block 固定大小，存一张图像的原始数据
Index 区N × 128B每个 Block 对应一条索引记录（车牌、时间、大小、CRC）
设计考量：Index 区放在文件末尾而非 Header 之后，因为 Block 区写入频率远高于 Index 区。将热数据放在一起，减少磁头跳跃。

文件头 (Header) — 4KB

字段类型大小说明

块大小选择

方案 A（推荐）：固定块 = 最大图像尺寸 × 安全系数

BlockSize = 3MB × 1.2 ≈ 3.6MB → 向上取整到 4MB（与 NTFS 大簇对齐友好）

每张图像固定占用一个 Block。优点：实现极简，性能恒定。缺点：平均浪费约 30% 空间（工业场景可接受）。

关键算法

初始化（首次启动 / 文件不存在）

Initialize(file_path, block_count, block_size):
  1. total_size = 4KB + block_count × (block_size + 128)
  2. 创建文件 (CreateFile)
  3. 预分配磁盘空间（关键，防碎片）
     → 优先: SetFileValidData (瞬时，需特权)
     → 回退: SetFilePointerEx + 分块写零 (每次 64MB)
  4. 初始化 Header，写入 magic / block_size / block_count
  5. Index 区全部清零
  6. FlushFileBuffers

写入图像

WriteImage(data, size, vehicle_id):
  前提: size ≤ block_size

  1. 获取写锁 (SRWLock Exclusive)
  2. 计算偏移:
     data_offset  = 4KB + write_position × block_size
     index_offset = 4KB + block_count × block_size + write_position × 128
  3. 写入图像数据 → 写入 IndexEntry → 更新 Header
     (顺序: 数据 → 索引 → 文件头，确保崩溃可恢复)
  4. write_position = (write_position + 1) % block_count
  5. write_sequence += 1
  6. FlushFileBuffers
  7. 释放写锁

读取图像

ReadByVehicle(vehicle_id):
  1. 获取读锁 (SRWLock Shared)
  2. 遍历 Index 区 N 个条目，匹配 vehicle_id
  3. 对每个匹配: data_offset = 4KB + block_index × block_size
     读取 → 校验 CRC32 → 返回
  4. 释放读锁

环形覆盖语义

当 write_position 绕回时，该位置原有的 IndexEntry.sequence 非零说明有旧数据，直接覆盖即可——无需显式删除操作。数据新旧通过 sequence 大小判断：数字越大越新。

并发模型

角色操作频率并发策略
相机采集线程写入图像每车一次（高频）独占写锁
UI 显示线程读取最新图像按需（中频）共享读锁
分析/回放线程按车辆 ID 读取按需（低频）共享读锁
推荐 SRWLock：Windows 用户态读写锁，写互斥、读共享，性能优于 Mutex。

异常恢复

崩溃时机后果恢复方式
写数据途中当前 Block 数据不完整CRC 校验失败 → 标记该 Block 为无效
写 Index 途中Index 与数据不一致以 Index 为准，数据 Block 可被重新使用
写 Header 途中write_position 未更新扫描 Index 找到最大 sequence 的位置，重建 Header

恢复流程（每次启动执行）

Recover(file_path):
  1. 打开文件，验证 Header magic 和 CRC32
  2. 扫描全部 Index 条目:
     → 找到最大的 valid sequence → 重建 write_position
  3. 遍历所有声称 Valid 的条目:
     → 校验 data_crc32
     → CRC 不匹配 → 标记为空 (sequence=0, flags=Empty)
  4. 写回修正后的 Header

};struct ImageInfo {
    uint64_t sequence;          // 全局写入序列号

    uint32_t actual_size;       // 图像实际字节数

struct RingBufferStats {
    uint32_t block_size, block_count;
    uint32_t used_blocks, free_blocks;
    uint64_t write_sequence, write_position;

    uint64_t oldest_sequence, newest_sequence;
    bool     is_healthy;
    uint32_t extent_count;
    bool     is_contiguous;
    uint64_t largest_free_extent;
};

class ImageRingBuffer {

    /// @brief 初始化环形缓冲区组件

    /// @return RBF_OK 成功，其他为错误码
