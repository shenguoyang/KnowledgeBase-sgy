根据软件需求规格说明书和可选的初步影响分析，输出一份完整的**代码影响分析文档**。本 skill 利用 CodeGraph（基于 SQLite 的工作区符号知识图谱）进行独立、有据可查的影响分析。绝不盲信输入文档——每一条结论都必须经过 CodeGraph 对代码库的验证或反驳。

**输入**：软件需求规格（自由文本/结构化/markdown）+ 可选的初步影响分析。两者都可能不完整或模糊。

**输出**：一份结构化的**代码影响分析文档**，包含 6 个章节 + 附录。主文档名为 `impact-analysis.md`；若在分析过程中识别出多种可行实现方案，需额外输出 `multiple-solutions.md`。两者均写入与**需求规格文档同级**目录；若用户未提供规格文件路径，则回退至 当前的claude code的根目录下

---

## 核心原则

- **验证，而非信任。** 输入的需求规格可能有误、过时或一厢情愿。每一条"需要改什么"的主张都必须经过 CodeGraph 验证——而不只是确认。
- **超越需求文档。** 用 CodeGraph 找出需求撰写者不知道的代码（未列出的依赖、隐藏耦合、副作用路径）。
- **先横向，再纵向。** 定位主符号后，不要立即沿一条调用链深入。先扫描代码库中所有类似参数的传递模式——同一组件内、传递方向相同（gen→sensor）、生命周期相同的数据有哪些路径？每种路径就是一个可行方案。横向扫描完成后再对首选路径纵向深入。
- **暴露模糊点。** 当需求描述含糊时（如"更新 RPC 层"），生成具体的澄清问题——不要猜测。
- **严格分类。** 每个影响点都必须归类为：`must-modify`（必须修改）、`should-sync`（应同步修改）或 `needs-verification`（需验证）。不允许模糊处理。
- **仅显式调用。** 本 skill 绝不自动触发。用户必须显式指定（输入 skill 名称或明确要求做代码影响分析）。
- **只读分析。** 应用本 skill 时，绝对不允许修改代码库中的任何文件。所有工作仅限于读取代码、执行 CodeGraph 查询、生成分析文档并写入 `impact-analysis.md` 和 `multiple-solutions.md`。不修改源码、不修改头文件、不修改工程配置。

## 执行步骤

### 步骤 1：接收并评估输入

如果用户未提供变更名或任何输入，使用 **AskUserQuestion 工具**询问：

> "请提供以下信息：1）变更名称（kebab-case 格式，如 'add-dual-camera-support'）；2）需求规格说明；3）初步影响分析（如有）。"

从输入中提取：
- **变更名称**（如未提供则从描述中推导）
- **需求规格**（需要变更的完整描述）
- **初步影响分析**（如有；可能为空）

**输出目录推导**：

- 如果用户以文件路径形式提供了需求规格（如 `doc/迭代2/需求2/需求.md`），则输出目录定为该文件所在目录（`doc/迭代2/需求2/`）。
- 分析开始前声明："影响分析文档将写入 `<目录路径>`"。

**护栏**：没有需求规格绝不继续。如果用户提供的内容不可操作，用更具体的提示再次询问。

宣告："正在分析变更影响：`<变更名>`"

### 步骤 2：将需求拆解为原子命题

将需求规格分解为离散、可验证的原子命题。每条命题是一个关于"必须改什么"的单一声明。例如：

- "在 RPC InspectOperator 协议中增加 `cameraDualMode` 字段"
- "将双相机标定数据存入 SQLite 数据库"
- "更新检测流程状态机以处理两路相机流"

**输出编号列表**供用户确认：

```
## 提取的原子命题
1. [命题1]
2. [命题2]
...
N. [命题N]

以上是否准确反映了变更意图？（确认/修改）
```

**护栏**：如果某条命题含混不清，立即标记并生成澄清问题。命题未经确认不进入下一步。

---

### 步骤 3：基于 CodeGraph 的全局代码分析

首先执行 `codegraph_status` 确认索引是最新的。

对每条确认的命题，按以下子阶段进行系统化的 CodeGraph 调查。独立的子阶段可并行执行。

#### 阶段 3-pre：多路径横向扫描（必须在纵向深入之前执行）

对每条原子命题涉及的核心数据（如"一个 bool 标志从 gen 传到 sensor 算法层"），**先扫描至少 2 条可抵达目标的独立调用链**，再选择首选路径纵向深入。

**方法**：
1. 识别"同类数据"：在当前目标组件中，用 `codegraph_search` 找出类型相同或语义相近的参数——例如要传 bool 标志到算法，搜索 sensor 组件中已有的 bool 参数是如何到达 `process()` 的
2. 对每条候选路径用 `codegraph_trace` 快速勾勒调用链
3. 对比各路径的改动面、持久化方式、生命周期

**常见多路径场景**：
- 数据经 DB 持久化（Teach 写 → Inspect 读）vs. 运行时 RPC 下发（Inspect 时实时传）
- 数据挂在已有结构体上（如 `InspectionParam`）vs. 新增独立 RPC 方法
- 数据存储在 gen 侧（每次查询）vs. 缓存在 sensor 侧（`switchPlanColor` 时下发）

**护栏**：每条原子命题至少找到 1 条备选路径。如只找到 1 条，显式声明"未发现替代路径"并说明搜索范围。横向扫描结果直接作为步骤 4.5 多方案识别的输入。

#### 阶段 3a：横向发现（跨层影响）

对每条命题，使用 `codegraph_context` 识别跨 4 层架构的所有潜在受影响符号：

1. **定位主符号**：用 `codegraph_context`，以命题描述作为任务输入
2. **跨层扩散**——追踪主符号的连接关系：
   - **应用层**：哪些应用（`inspection_client`、`inspection_server`、`specular_sensor_server`、`data_docking_service`、`worker_screen_client`、`defect_data_tool`）使用了该符号？
   - **组件/业务层**：哪些 DLL（`specular_vision_gen`、`specular_vision_sensor`、`auto_control`、`robot_control`、`pillar_lamp_control`）包含或调用了该符号？
   - **RPC 层**：变更是否触及 `rpc_common` 协议头文件或 `rpc_client` 代理？
   - **FPGA/硬件层**：变更是否触及 `fpga_adapter`、`fpga_control` 或寄存器级代码？

用 `codegraph_callers` 枚举每个主符号的消费者，用 `codegraph_callees` 枚举每个主符号的依赖项。

#### 阶段 3b：纵向发现（调用深度）

对每个主符号，用 `codegraph_trace` 追踪从顶层入口点到叶子实现的完整调用路径。

**必查的关键路径**：
- `*Operator::*`（RPC 入口）→ `*Impl::*`（实现）→ `*Dao::*`（数据库）或 `*Adapter::*`（FPGA）
- `*Manager::*`（业务逻辑）→ `*Dao::*`（持久化）
- 状态机转换：`StateMachine` → `State*` → `Flow*`

#### 阶段 3c：反向依赖分析（影响半径）

对每个被标记为需要变更的符号，使用 `codegraph_impact`（depth 最低为 2）确定：
- 谁调用了此符号（上游——签名/行为变更后谁会崩溃）
- 此符号调用了谁（下游——哪些副作用会级联）
- 跨组件依赖：`specular_vision_sensor` 的变更是否会传播到 `specular_vision_gen` 或 `inspection_client`？

**护栏**：每个 `must-modify` 符号必须执行 depth >= 2 的 impact 分析。协议或 schema 变更使用 depth=3 或更高。

#### 阶段 3d：时间轴维度分析

将每个影响点按运行模式分类：
- **仅 Teach 模式**：标定、配置、训练流程
- **仅 Inspect 模式**：实时检测流水线
- **两者皆有**：Teach 中配置/设置，Inspect 中消费

用 `codegraph_trace` 从 teach 入口点（如 `TeachOperator::pmdStart`）和 inspect 入口点（如 `InspectOperator::enter`）追踪，判断哪些路径重叠。

**实时路径特殊处理**：只要影响点落在 inspect 路径上，自动标注 `performance` 风险。

---

### 步骤 4：非代码资产发现

对非代码资产进行针对性搜索：

#### 4a：RPC 协议定义

用 `codegraph_search` 搜索受影响的协议头文件：
- 查询模式：`*_RPC_*`、`*_PARAM_OBJECTS`，找到所有宏生成的 RPC 参数块
- 读取三个关键协议头文件，检查是否断裂：
  - `components/rpc_common/specular_vision_gen/specular_vision_gen_rpc_json.h`
  - `components/rpc_common/specular_vision_sensor/specular_vision_sensor_rpc_json.h`

#### 4b：数据库 Schema

搜索 DAO 类及字段对象宏：
- `codegraph_search` query=`*Dao` kind=class
- `codegraph_search` query=`*_DAO` kind=class
- `codegraph_search` query=`DATA_BASE_*_FIELD_*` kind=constant

将每个 DAO 映射到其表名（从 DAO 头文件中的 `#define *_DATA_BASE_NAME` 模式提取）。

#### 4c：配置文件

搜索配置管理相关符号：
- `codegraph_search` query=`SystemConfigMgr`
- `codegraph_search` query=`*ConfigMgr`
- `codegraph_search` query=`json::Value`

#### 4d：构建系统

枚举包含受影响源文件的 `.vcxproj` 和 `.sln` 文件：
- `codegraph_files` pattern=`**/project/**/*.vcxproj`
- 与受影响的源文件交叉比对，确定哪些工程需要更新

### 步骤 4.5：多方案识别

**注意**：多方案识别不是等到 Step 4 才开始的——阶段 3-pre 的横向扫描已经发现了候选路径。Step 4.5 的工作是**整理和对比**这些路径，决定是否需要生成独立的 `multiple-solutions.md`。

在分析过程中，如果发现同一需求可以通过**两种或以上**明显不同的技术路径实现，必须生成一份**多方案对比文档** `multiple-solutions.md`，写入与 `impact-analysis.md` 同级的输出目录。

**识别触发条件**（满足任一条即触发）：

| 触发条件 | 示例 |
|---------|------|
| 数据库层存在多种迁移策略 | 直接修改现有表 vs. 新建配置表；ALTER TABLE vs. 新建表+数据迁移 |
| RPC 协议存在兼容 vs. 破坏两种设计 | 新增可选字段 vs. 修改必选字段语义 |
| 架构层实现位置存在争议 | 功能可在 `specular_vision_gen` 实现，也可在 `inspection_server` 实现 |
| 硬件层存在抽象 vs. 直接操作两种路径 | 通过 FPGA 适配器抽象 vs. 直接写寄存器 |
| 同一问题存在不同重构深度 | 最小改动（就地修补）vs. 深度重构（引入新抽象层） |
| 同一数据可通过 DB 持久化传递，也可通过运行时 RPC 传递 | bool 标志可存 MeasurePoint 表（Teach 时写），也可通过 switchPlanColor（Inspect 时传） |

### 步骤 5：影响分类

将每个发现的影响点归入三种类别之一：

| 分类 | 判定标准 | 示例 |
|------|---------|------|
| **must-modify**（必须修改） | 不改此代码，变更完全无法工作 | 在 RPC 参数块中新增字段，且该字段被实现代码读取 |
| **should-sync**（应同步修改） | 不改也能工作，但系统会不一致或变得更脆弱 | 更新客户端 mock/stub 以匹配服务端新增的 RPC 参数 |
| **needs-verification**（需验证） | 影响不确定，需要测试或深入调查才能确认 | 某条极少使用的代码路径可能通过函数指针调用了被修改的函数 |

**缺陷发现姿态**：分类过程中，用 `codegraph_callers` 和 `codegraph_callees` 查找以下情况：
- 被修改的参数在某处被消费，但初步分析中未列出对应的调用方
- 共享结构体被修改，但遗漏了某个消费者
- 宏生成的定义传播到了需求撰写者未知的代码

---

### 步骤 6：风险标注

为每个 `must-modify` 和 `should-sync` 影响点标注适用风险标签：

| 风险标签 | 适用条件 | 检测方法 |
|----------|---------|---------|
| **[T] 线程安全** | 代码使用 `std::mutex`、`std::atomic`、`QThread`，或被多个线程调用 | `codegraph_search` 搜索文件中的 mutex/atomic；检查符号是否被工作线程调用 |
| **[P] 性能** | 代码位于 inspect 实时路径上（图像以生产节拍流经此处） | 检查 `codegraph_trace` 从 `InspectOperator::enter` 或 `FlowInspect` 是否能到达此代码 |
| **[C] 兼容性** | RPC 协议参数块被修改（客户端和服务端必须一致） | 检查文件是否在 `rpc_common/` 下，或其符号是否经由 `param_object.h` 宏序列化 |
| **[H] 硬件** | 代码触及 `fpga_control.h`、`fpga_adapter.h` 或任何寄存器级 I/O | 检查文件路径及符号名是否含 `fpga`、`register`、`mmap`、`ioctl` |
| **[M] 迁移** | 数据库表 schema 被修改（新增/删除列，新增/删除表） | 检查 DAO 文件中 `FIELD_*` 宏的变更；检查增删表的 `DataBaseHelper` 调用 |

**允许多标签叠加。** 一个影响点可能同时携带多个风险标签。

---

### 步骤 7：生成澄清问题

当输入的需求规格模糊、不完整或自相矛盾时，生成针对性的澄清问题。按以下决策树：

1. **缺失细节**："需求提到'更新数据库'但未指明具体哪些表。我在受影响组件中发现了 N 个 DAO 类。请问涉及哪些表？"
2. **与代码库矛盾**："需求假设存在类 `X`，但 CodeGraph 显示实际使用的是 `Y` 模式。应该沿用现有模式还是引入新抽象？"
3. **未声明的副作用**："修改结构体 `X`（被 N 个 DAO 和 M 个 RPC 方法使用）会产生级联影响。这是预期内的还是应该引入独立的结构体？"
4. **范围模糊**："需求提到'FPGA 初始化'但未区分抽象层和寄存器级控制。变更目标是哪一层？"
5. **时间轴模糊**："此变更仅用于 teach 模式、仅用于 inspect 模式，还是两者都需要？"

**护栏**：不要对需求已明确回答的问题进行提问。只有当答案会实质性地影响分析时才提问。

## 护栏

### 必须做
- 分析开始前运行 `codegraph_status` 确认索引是最新的
- 用 `codegraph_context` 作为每条命题的**首要**发现工具——它一次调用组合了 search + node + callers + callees
- 用 `codegraph_trace` 验证符号间的调用路径——绝不因为头文件 `#include` 了就假设依赖存在
- 对每个 `must-modify` 符号执行 depth >= 2 的 `codegraph_impact`
- 将初步影响分析中的每条主张与 CodeGraph 交叉验证——矛盾点明确记录在第 5 章和附录中
- 即使需求只提到一个协议头文件，也必须检查全部三个 RPC 协议头文件
- 检查受影响组件 `data/` 目录下的全部 DAO 类
- 为每个影响点标注时间轴模式（teach/inspect/两者）
- 为 `must-modify` 和 `should-sync` 项标注风险标签
- 当分析涉及多个可行修改方案时（见步骤 4.5 触发条件），必须输出 `multiple-solutions.md` 到与 `impact-analysis.md` 同级目录
- 将输出文档写入需求规格文档同级目录（优先使用输入规格文件的所在目录；无文件路径时回退至 `openspec/changes/<变更名>/` 下）。主文档名为 `impact-analysis.md`；多方案时同时输出 `multiple-solutions.md`

### 禁止做
- 不得未经 CodeGraph 验证就信任初步影响分析
- 不得跳过反向依赖分析（影响半径）
- 不得在无 CodeGraph 证据的情况下将影响点标为 `must-modify`
- 不得跳过非代码资产（数据库、RPC 协议、配置、构建系统、版本号）
- 不得在没有确认需求规格的情况下越过步骤 1
- 不得撰写空泛的影响分析——每条记录必须引用 CodeGraph 找到的具体文件和符号
- 不得在输入含混时跳过澄清问题章节
- 不得自动触发——本 skill 必须由用户显式调用
- 不得将多方案分析直接塞进 `impact-analysis.md` 而不另建 `multiple-solutions.md`
- 不得在无输入文件路径时推测输出目录——必须回退到 `openspec/changes/<变更名>/`
- **不得修改代码库中的任何文件**——本 skill 为纯只读分析，禁止修改源码、头文件、工程配置或任何其他仓库内文件

### 边界情况
- **初步分析为空**：正常——本 skill 完全基于 CodeGraph 从零发现。正常推进。
- **需求提到的符号 CodeGraph 未找到**：在第 5 章中标记为矛盾点。符号可能拼写错误、尚未创建、或位于未索引文件中。
- **影响半径过大（>50 个影响点）**：询问用户是缩小范围还是继续完整分析。建议将变更拆分为更小的子变更。
- **协议变更破坏向后兼容性**：在风险汇总中标记为"严重"。在第 3.2 节中生成专门的兼容性分析。
- **数据库 schema 变更缺少迁移路径**：在第 5 章中标记为阻塞问题。不要假设迁移策略。
- **变更同时触及 teach 和 inspect 模式且有不同影响**：按模式拆分影响分析表，或在第 2 章中增加专门的时间轴分析子章节。
- **调用图追踪遇到动态分发断点**：当 `codegraph_trace` 报告动态分发断点（Qt 信号/槽、函数指针）时，显式标注断点位置，并用 `codegraph_node` 查看源码以找到下一跳。
- **分析过程中发现多个可行方案**：触发步骤 4.5，生成 `multiple-solutions.md`。主文档仍按首选方案编写。若用户在新方案中提供了额外约束，需要重新评估首选方案并对比更新。
- **输出目录与 openspec 结构冲突**：需求规格与 `openspec/changes/<变更名>/` 不在同一目录时，优先选择需求规格同级目录。若确实需要按传统 openspec 结构存档，由用户显式指定。

完整的路径模式→内容映射见代码库索引文档：[.claude/codebase-index.md](../../codebase-index.md)，其中包含应用层、组件/业务层、RPC 层、FPGA/硬件层、数据库层、构建系统、测试等全部路径约定和速查表。

## 工具使用速查

| 分析目标 | CodeGraph 工具 | 备注 |
|---------|---------------|------|
| 理解某个功能区域 | `codegraph_context` | **首要工具**——单次调用组合了 search + node + callers + callees |
| 追踪调用路径 A → B | `codegraph_trace` | 单次调用返回完整路径（含动态分发跳转） |
| 查找修改后会破坏什么 | `codegraph_impact` | 始终 depth >= 2 |
| 查找符号的所有调用方 | `codegraph_callers` | 需要完整调用方列表时使用 |
| 查找符号调用了谁 | `codegraph_callees` | 追踪下游依赖时使用 |
| 读取符号源码 | `codegraph_node` | 使用 `includeCode: true` |
| 浏览多个符号 | `codegraph_explore` | 上限可控——一次返回若干相关符号源码 |
| 按名称查找符号 | `codegraph_search` | 用于协议模式、DAO 名称、版本宏 |
| 浏览目录结构 | `codegraph_files` | 用于构建系统和配置文件发现 |
| 检查索引新鲜度 | `codegraph_status` | 分析开始时运行以确认索引最新 |

# 重要

### 验证，而非假设
当修改一个跨层（数据结构 → RPC → 数据库 → 用户界面）的数据结构或行为时，不要止步于找到第一个匹配项。一个头文件中的结构体改动，往往需要在序列化、持久化存储和用户界面这三个层面进行相应的修改——每一层使用的机制都不同。在断定修改已完成之前，请追踪数据的完整生命周期。
