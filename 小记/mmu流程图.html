1 CPU 发出虚拟地址 (Virtual Address)
CPU 执行指令，生成 48 位虚拟地址
触发源

当 CPU 执行 mov rax, [0x7f8a3c00b000] 时，该虚拟地址被送入 MMU。

虚拟地址 (48-bit) 0x00007f8a3c00b000
字段说明本例值
VPN (Virtual Page Number) 虚拟页号，用于索引页表 0x7f8a3c00b (36 bits)
VPO (Virtual Page Offset) 页内偏移，直接复制到物理地址 0x000 (12 bits)
关键认知： 虚拟地址 = VPN + VPO。VPN 负责找到物理页框，VPO 是页内具体位置。
4KB 页 → VPO 占 12 位 (2¹² = 4096)。VPN 占剩余 36 位。
2 MMU 拆分虚拟地址为四级索引
将 36 位 VPN 拆成 4 个 9 位索引，每级 512 项
地址拆解

在 x86-64 4级页表中，MMU 把 36 位 VPN 等分为 4 段，每段 9 位（2⁹ = 512 项），逐级索引。

PM L4 索引 VL4
bits 47:39 (9位)
+
PM L3 索引 VL3
bits 38:30 (9位)
+
PM L2 索引 VL2
bits 29:21 (9位)
+
PM L1 索引 VL1
bits 20:12 (9位)
+
页内偏移 VPO
bits 11:0 (12位)
为什么 4 级？ 4 级 × 每级 512 项 = 可管理 256TB 虚拟地址空间。
如果某级页表项标记为 大页 (2MB/1GB)，遍历可提前结束，省去后续级别。
⚡ TLB 缓存命中率极高 — 绝大多数转换跳过全部 4 级遍历，直接在 TLB 中得到结果。
3 TLB 快表查询 (Translation Lookaside Buffer)
TLB = VPN → PPN 的硬件缓存，命中则 1 周期完成
硬件缓存

TLB 是一个小型、全相联或组相联的硬件缓存，存储最近使用的 VPN→PPN 映射。它是 MMU 内部的 SRAM，访问极快。

ITLB + DTLB 架构 (以 Skylake 为例)

TLB 层级用途条目数页大小延迟
L1 ITLB指令取指128 项 (8路组相联)4KB1 周期
L1 DTLB数据读写64 项 (4路组相联)4KB1 周期
L2 STLB统一二级1536 项 (12路)4KB / 2MB~8 周期

TLB 查找流程

// 硬件并行比较：VPN 与 TLB 所有条目的 Tag for (each entry in TLB) { if (entry.tag == VPN && entry.valid) { // ✅ TLB HIT — 1 周期 return { PPN: entry.ppn, flags: entry.flags }; } } // ❌ TLB MISS — 需走页表遍历

TLB Hit ~1 周期，99%+ 的转换命中

TLB Miss ~20-100 周期，触发硬件页表遍历

TLB 本质上是一个 内容可寻址存储器 (CAM)，允许并行比较所有条目。

4 硬件页表遍历 (Page Table Walk)
TLB Miss → MMU 硬件自动走 4 级页表 (x86-64)
核心流程

当 TLB 未命中时，MMU 从 CR3 寄存器（存放 PGD 物理基址）出发，逐级读取页表。

每一步做了什么？

步骤输入操作输出
① CR3 + VL4 读取 PGD[VL4] = PGD基址 + VL4 × 8 字节 PUD 基址 (40位物理地址)
② PUD基址 + VL3 读取 PUD[VL3] = PUD基址 + VL3 × 8 字节 PMD 基址
③ PMD基址 + VL2 读取 PMD[VL2] = PMD基址 + VL2 × 8 字节 PTE 数组基址
④ PTE基址 + VL1 读取 PTE[VL1] = PTE基址 + VL1 × 8 字节 PPN 物理页框号
⚠️ 每级都需访存一次 — 4 级页表 = 额外 4 次内存访问！
这就是 TLB 如此重要的原因：把 4 次访存缩减为 1 次 TLB 查找。

6 缺页异常 (Page Fault) — Present 位为 0 时
硬件触发 #PF 异常 → 操作系统接管处理
异常路径

当页表遍历中某级的 P (Present) 位 = 0，说明该页不在物理内存中。MMU 触发 14 号异常 (Page Fault)，操作系统介入。

// CPU 硬件动作 1. 保存故障地址到 CR2 寄存器 2. 将错误码压栈 - bit0: 0=缺页, 1=保护违例 - bit1: 0=读, 1=写 - bit2: 0=内核, 1=用户态 - bit3: 1=保留位违例 - bit4: 1=取指 3. 跳转到 IDT[14] → page_fault_handler()
// Linux 内核处理 (do_page_fault) switch (fault_type) { // 情况1: 合法缺页 case VALID_ACCESS: page = alloc_page(); // 分配物理页 read_from_disk(page); // 从磁盘读入 update_page_table(page, pte); // 更新PTE return; // 重试指令 ✅ // 情况2: 非法访问 (如写只读页) case WRITE_TO_READONLY: send_signal(SIGSEGV); // 段错误 ❌ }
缺页不一定是错误： 大多数缺页是正常的"按需调页 (Demand Paging)"——
程序首次访问某页，内核才从磁盘加载。这也是 mmap 文件映射、写时复制 (Copy-on-Write) 的底层机制。
7 组装最终物理地址
物理地址 = PPN << 12 | VPO
最终输出

从 TLB 或页表遍历拿到 PPN 后，MMU 将 VPO（页内偏移，低12位）直接拼接到 PPN 后面。

物理页框号 PPN 来自 TLB / PTE
高位 (物理地址宽度 − 12)
∥
页内偏移 VPO 直接复制
低 12 位
物理地址 = (PPN << 12) | VPO
VPO 不变！页是映射的最小单位，偏移不需要转换
VPO (12位) = PPO (12位)，因为页大小为 4KB = 2¹² 字节。
页内偏移在虚拟和物理地址中完全相同，无需转换。

性能总结
场景TLB页表遍历Cache总延迟

常规路径 L1 TLB Hit (1c) 无需 L3 Cache Hit (~36c) ~37 周期
TLB Miss Miss → STLB Hit (8c) 无需 L1 Hit (~4c) ~12 周期

缺页异常 Miss 遍历 + 异常处理 磁盘 IO ~10ms (百万周期)
关键结论： TLB 是现代 CPU 性能的基石。TLB 命中时，地址转换几乎"免费"。
TLB Miss 的代价是几十倍，而 Page Fault 则是上百万倍的惩罚 —— 这就是为什么所有操作系统都会尽量让热数据保持在内存中。
