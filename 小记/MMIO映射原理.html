完整追踪：从 PCI 总线上的物理设备，到进程地址空间中的虚拟地址

一、什么是 MMIO（内存映射 I/O）

CPU 要和外部设备（显卡、NVMe 硬盘、网卡、USB 控制器等）通信，有两种方式：

Port I/O（端口 I/O）

使用独立的 I/O 地址空间，CPU 通过 IN/OUT 专用指令访问。

历史遗留方式，x86 上仍有少量使用（PS/2、PIT 等），但已逐渐被 MMIO 取代。

MMIO（内存映射 I/O）

把设备的寄存器直接映射到物理地址空间，CPU 用普通访存指令 MOV 就能读写设备。

现代设备的主流方式，简洁高效，可以利用 CPU 的全部寻址能力。

核心思想：设备的一组寄存器（控制寄存器、状态寄存器、数据缓冲区），在 CPU 看来就是一段"特殊的内存地址"。读这个地址 = 读设备寄存器，写这个地址 = 写设备寄存器。

类比：想象一栋大楼（CPU 的地址空间），大部分房间是 RAM（普通内存），但有几个房间的门后面不是 RAM，而是外部设备的控制面板。你拧开房门（发起内存访问），实际操作的是设备而不是内存。

为什么用 MMIO？

  • 统一编程模型：不需要特殊的 I/O 指令，用 C 指针就能访问设备
  • 更大的地址空间：Port I/O 只有 64KB 空间，MMIO 可以使用整个物理地址空间
  • 更高的带宽：可以利用 CPU 的 cache line 宽度、burst 传输等机制
  • 与虚拟内存自然结合：通过页表可以控制访问权限、进行地址转换

二、物理地址空间布局

CPU 发出的每一个"地址"，并不一定都到达 RAM（内存条）。CPU 的地址总线连接到系统代理 / 北桥，由它决定一个物理地址对应的是RAM还是MMIO 设备。

现代 x86 平台的典型地址划分（48 位物理地址 = 256TB）

物理地址范围大小目的地说明
0x0 – 0x9FFFF 640KB DRAM 传统 DOS 可用内存
0xA0000 – 0xBFFFF 128KB VGA 显存 传统显卡帧缓冲区（legacy 窗口）
0xC0000 – 0xFFFFF 256KB BIOS ROM BIOS/UEFI 固件映射
0x100000 – TOLUD 变化 DRAM 常规系统内存（到 TOLUD 为止）
TOLUD – 4GB ~0.5-1.5GB MMIO 设备 32 位 MMIO 窗口（PCI BAR、APIC、HPET 等）
4GB – 上限 变化 DRAM 高于 4GB 的系统内存
高地址区域 变化 MMIO 设备 64 位 MMIO 窗口（大 BAR 设备在此）
TOLUD（Top of Low Usable DRAM）：由 BIOS 设定，标记 32 位地址空间中 DRAM 的终点。通常是 2GB~3.5GB 之间的某个值。TOLUD 以上到 4GB 留给 MMIO。

为什么要在 4GB 以上再做 DRAM：因为 32 位地址空间太拥挤（只有 4GB），必须把部分 DRAM 重映射到 4GB 以上的地址空间，把 4GB 以下的宝贵空间留给 MMIO 设备。

地址路由：CPU 发出的物理地址怎么找到对的地方

CPU 发出 物理地址
→
系统代理 (SA) 地址解码器
→
DRAM 控制器 去内存条
或
PCIe 根端口 去设备总线

系统代理内部有一个地址解码器（Address Decoder），它维护了一个地址范围路由表。每当 CPU 发起一个物理地址访问，它查表决定：

  • 这个地址落在 DRAM 的范围 → 路由到内存控制器 → 去内存条
  • 这个地址落在 MMIO 的范围 → 路由到 PCIe 根端口 → 去 PCIe 总线 → 最终到达设备
关键点：这个路由表是由 BIOS/UEFI 在启动时配置的。BIOS 先探测所有设备，计算每个设备需要多少地址空间，然后统一分配物理地址范围，最后把路由规则写入 CPU 内部的地址解码寄存器。

三、设备发现：PCI/PCIe 总线枚举

一个设备插入 PCIe 插槽后，系统需要知道"这是个什么设备、需要多少地址空间"。这个发现过程叫 PCI 枚举。

3.1 PCI 配置空间

每个 PCI/PCIe 设备都有一块独立的"身份证和需求表"——配置空间（Configuration Space），大小 4KB。

配置空间中最重要的几个字段：

  • Vendor ID / Device ID（偏移 0x00/0x02）：标识"我是谁"（如 NVIDIA、Intel）
  • Class Code（偏移 0x0B）：设备类型（网卡、显卡、存储控制器等）
  • BAR0 ~ BAR5（偏移 0x10 ~ 0x27）：最关键！六个 Base Address Register

3.2 BAR（Base Address Register）—— 设备的"地址需求单"

每个 BAR 是一个 32 位或 64 位的寄存器，包含两个信息：

  1. 地址空间大小：设备需要多少字节的 MMIO 空间（或被写入的起始地址）
  2. 属性位：是内存映射还是 I/O 端口？64 位还是 32 位？是否可预取？
1 BAR 寄存器结构（32 位 BAR 示例） 
Bit  0      : 0 = 内存映射, 1 = I/O 端口
Bits 2-1    : 00 = 32 位地址, 10 = 64 位地址
Bit  3      : 可预取（Prefetchable）
Bits 31-4   : 基地址 / 大小（由低位是否为 0 决定）
2 探测 BAR 大小的方法

系统怎么知道一个 BAR 需要多少空间？经典的三步法：

// Step 1: 读取 BAR 的当前值（假设 BIOS 已经分配了一个临时地址）
old = read_config(dev, BAR0);

// Step 2: 向 BAR 写入全 1 (0xFFFFFFFF)
write_config(dev, BAR0, 0xFFFFFFFF);

// Step 3: 读回来。硬件只认低位中不可改的位为 0
//         举例：读回来是 0xFFF00000
//         说明低 20 位全是 0 → 大小 = 2^20 = 1MB
size_raw = read_config(dev, BAR0);
size = ~(size_raw & 0xFFFFFFF0) + 1;  // 按位取反 +1 = 1MB

// Step 4: 恢复原值
write_config(dev, BAR0, old);

原理：BAR 中代表地址的低若干位如果被硬件强制为 0，那么写 1 进去也读不回 1。这些"写不进 1"的位就暴露了设备需要多大的对齐空间。

3.3 枚举完整流程

1 BIOS 扫描 PCI 总线

BIOS 遍历 Bus 0~255, Device 0~31, Function 0~7，逐个读取配置空间。如果 Vendor ID ≠ 0xFFFF，说明该位置存在设备。这叫做 PCI 总线枚举（Bus Walk）。

2 读取设备需求

读取设备的 BAR0~BAR5，通过"写全 1"的方法测出每个 BAR 需要的地址空间大小。同时记录每个 BAR 的属性（是否 64 位、是否可预取等）。

3 统一分配物理地址

BIOS 汇总所有设备的 BAR 需求，像一个"内存分配器"一样，在物理地址空间中找到合适的区间分配给每个 BAR。32 位 BAR 只能分配在 4GB 以下的 MMIO 窗口，64 位 BAR 可以分配在 4GB 以上。

分配完成后，BIOS 将分配的起始物理地址写入每个 BAR 寄存器。

4 配置地址解码器

BIOS 把最终的路由规则写入 CPU 系统代理（或芯片组）的地址解码寄存器，告诉硬件："物理地址范围 0xFE000000~0xFE0FFFFF 去 PCIe 总线上的设备 X"。

5 写入 ACPI 表

BIOS 把分配结果写入 ACPI MCFG 表和 DSDT 中的 _CRS 方法，供操作系统内核启动后读取。

3.4 PCIe 扩展：ECAM（增强配置访问机制）

传统的 PCI 配置空间通过 0xCF8/0xCFC I/O 端口访问（慢）。PCIe 引入了 ECAM：把每个设备的 4KB 配置空间直接映射到 MMIO 地址。

ECAM 映射公式
MMIO地址 = MCFG基址 + (Bus << 20) + (Device << 15) + (Function << 12) + 偏移

例如：MCFG 基址 = 0xC0000000，访问 Bus 0 / Dev 2 / Func 0 / 偏移 0x10（BAR0）
→ 实际地址 = 0xC0000000 + (0 << 20) + (2 << 15) + (0 << 12) + 0x10 = 0xC0020010
CPU 对这个地址做 MOV 读，就能直接读到该设备的 BAR0 值！

四、ACPI 和 Device Tree：非 PCI 设备的发现

并不是所有 MMIO 设备都挂在 PCI 总线上。CPU 内部的 APIC（高级可编程中断控制器）、HPET（高精度定时器）、UART 串口等，它们的 MMIO 地址是固定或由固件配置的。

ACPI（x86/x64 平台）

BIOS 启动时构造一套 ACPI 表（存在内存中），操作系统内核启动后去解析这些表：

  • MCFG：PCIe ECAM 基址
  • MADT / APIC：Local APIC 的 MMIO 地址
  • HPET：高精度定时器的 MMIO 地址
  • DSDT / SSDT：通过 _CRS 方法描述设备的 MMIO（和 I/O）资源

ACPI 是 BIOS 给 OS 的"交接清单"，告诉 OS 系统中存在哪些硬件、它们各自占用了哪些资源。

Device Tree（ARM/RISC-V 平台）

设备树是一个树形数据结构（.dtb 文件），由 Bootloader（如 U-Boot）传递给内核。节点中用 reg 属性描述 MMIO 地址范围：

uart0: serial@1c28000 {
    compatible = "snps,dw-apb-uart";
    reg = <0x01c28000 0x400>;
    interrupts = <0 1 4>;
};

意思：UART 的寄存器映射在物理地址 0x01c28000，范围 0x400 字节。

无论 ACPI 还是 Device Tree，内核解析后都会得到一个统一的"资源列表"：每个设备的物理地址范围和属性。这是后续 ioremap 的基础。

五、内核映射：ioremap() 的完整过程

此时，设备已经被 BIOS 分配了物理地址，设备寄存器已经可以被物理地址访问了。但操作系统内核运行在虚拟地址空间，不能直接用物理地址去访问设备——它必须创建从虚拟地址到物理地址的映射。

5.1 为什么内核需要虚拟地址

  • 内核开启了分页机制（MMU），所有访存指令使用的都是虚拟地址
  • 直接使用物理地址会导致 page fault
  • 通过页表映射，还可以精细控制这段映射的属性（比如设备寄存器绝对不能缓存）

5.2 设备驱动中的典型代码

// Linux 驱动中映射 MMIO 的典型模式
// phys_addr: PCI BAR 给出的物理地址（如 0xFEB00000）
// size:      BAR 测出的大小（如 0x1000 = 4KB）

// 1. 先"保留"这段物理地址区域（告诉内核：这里不是 RAM，别再分配给别人）
struct resource *res = request_mem_region(phys_addr, size, "mydev");

// 2. 创建虚拟映射：ioremap 的核心！
void __iomem *vaddr = ioremap(phys_addr, size);

// 3. 现在就可以像访问内存一样访问设备寄存器了
writel(0x1234, vaddr + REG_OFFSET);  // 写寄存器
uint32_t val = readl(vaddr + REG_OFFSET); // 读寄存器

5.3 ioremap() 内部做了什么

ioremap() 不是"魔法"，它底层做的就是修改页表。让我们逐层拆解：

1 在 vmalloc 区域找到一块空闲的虚拟地址

内核有一个专门的区域叫 vmalloc 区域，用于动态映射。ioremap 从这个区域中分配一段虚拟地址（比如 0xFFFF C900 0000 0000）。

2 逐页创建页表条目（PTE）

对于这段虚拟地址范围内每一页（4KB），在内核页表中创建一个 PTE：

  • 物理页框号 = phys_addr >> 12（物理地址的高位部分）
  • Present 位 = 1（该页存在）
  • Read/Write 位 = 1（可读写）
  • PCD 位 = 1（禁用缓存！设备寄存器不能缓存）
  • PWT 位 = 1（写穿模式！写入必须立刻到达设备）
  • XD 位 = 1（不可执行，设备寄存器不能当作代码运行）
3 刷新 TLB

修改了页表后，必须刷新 TLB，确保 CPU 下次访问这个虚拟地址时走新的页表条目。

5.4 关键：MMIO 映射必须是非缓存的（Uncacheable）

为什么设备寄存器绝对不能缓存？

假设 CPU 读了一个设备状态寄存器（值=0x00，表示"设备空闲"）。如果缓存了，下次 CPU 再读同一个地址，它可能直接从 L1 缓存中拿到旧值 0x00——此时设备实际状态已经变成了 0x01（"数据就绪"）。CPU 就会永远等待一个已经就绪的数据。

同样，写操作如果被缓存了延迟写入，设备可能收不到命令。所以 MMIO 区域必须设置为 UC（Uncacheable） 或 WC（Write-Combining，用于显存）。

x86 的缓存控制机制

  • MTRR（Memory Type Range Registers）：CPU 内部的寄存器，由 BIOS 设置。为物理地址范围设定缓存类型。比如"0xFEC00000~0xFECFFFFF 是 APIC，设为 UC"。
  • PAT（Page Attribute Table）：页表条目中的 PCD/PWT/PAT 三位组合，在 MTRR 的基础上再精细控制。
  • 最终类型 = MTRR 和 PAT 的组合（取最严格的限制）。
PATPCDPWT内存类型适用场景
000WB（写回）普通 RAM
001WT（写穿）帧缓冲区（显存写穿）
010UC-（不缓存）MMIO 设备寄存器
011UC（强不缓存）APIC、HPET 等固定硬件
100WC（写合并）显存（GPU 帧缓冲）
101WP（写保护）很少用

六、页表机制详解：虚拟地址如何翻译为物理地址

前面的 ioremap 设置了页表，但 CPU 在实际执行 mov [vaddr], eax 时，如何一步步从虚拟地址走到物理设备？

6.1 x86-64 四级页表结构

CR3 寄存器 存 PML4 物理地址
→
PML4 表 512 条目，每个指一个 PDPT
→
PDPT 表 512 条目，每个指一个 PD
→
PD 表 512 条目，每个指一个 PT
→
PT 表 512 条目，PTE 指向最终物理页
→
物理设备 MMIO 寄存器

6.2 逐步翻译示例

假设 ioremap 返回虚拟地址 0xFFFF C900 0001 0000，对应的物理地址是 0xFEB0 0000（一个 NVMe 设备的 BAR0）。

虚拟地址：0xFFFF C900 0001 0000 (64 位)

二进制拆分（x86-64 使用 48 位虚拟地址，高位全 1 或全 0）：

1111 1111 1111 1111  1100 1001 0000 0000  0000 0000 0000 0001  0000 0000 0000 0000
╵────────────┬──────╵ ╵──────┬───────╵ ╵──────┬───────╵ ╵──────┬───────╵ ╵───┬──╵
  Bit 63-48 未使用     PML4 索引   PDPT 索引    PD 索引     PT 索引   页内偏移
  (符号扩展)          (9 bits)    (9 bits)    (9 bits)   (9 bits)  (12 bits)
                      = 0x100     = 0x90      = 0x00     = 0x10    = 0x000

IX 步骤：
  1. CR3 → PML4 基址
  2. PML4[0x100] → PDPT 基址 (读取 8 字节，取物理地址 + 标志位)
  3. PDPT[0x090] → PD 基址
  4. PD[0x000]   → PT 基址
  5. PT[0x010]   → 最终物理页框 0xFEB00 (即地址 0xFEB0 0000)
     └─ PTE 中的标志：Present=1, R/W=1, PCD=1, PWT=1 (UC)
  6. 物理地址 = 0xFEB0 0000 + 0x000(页内偏移) = 0xFEB0 0000
整个过程由 MMU 硬件自动完成。CPU 发出虚拟地址 → MMU 查 TLB（有没有缓存翻译结果？）→ 没有就做上面这 4 步"页表遍历" → 得到物理地址 → 发给系统代理 → 系统代理看到 0xFEB00000 落在 PCIe 范围 → 路由到 PCIe 控制器 → 到达设备。

TLB 命中时，整个翻译只需 1 个时钟周期。TLB 未命中时，4 级页表遍历需要走 4 次内存读（或从缓存读，如果页表项在缓存中），约 10~30 个周期。

6.3 大页映射（Huge Page）

有些 MMIO 区域非常大（比如 GPU 显存可能需要数 GB 的 BAR 空间）。逐 4KB 页映射太浪费页表空间，可以用大页：

  • 2MB 大页：PD 条目直接指向 2MB 物理块（跳过 PT 级别）→ 页表省一级
  • 1GB 大页：PDPT 条目直接指向 1GB 物理块（跳过 PD 和 PT）→ 页表省两级

ioremap 通常使用 4KB 页，但 GPU 驱动的显存映射经常用大页以减少 TLB 压力。

七、用户态进程如何访问 MMIO

到目前为止，我们讲的映射都发生在内核空间——ioremap 修改的是内核页表。那用户态程序能不能也直接访问 MMIO 设备寄存器？

能，但需要特殊机制。主要有三条路径：

路径 1：/dev/mem + mmap（最原始的方式）

// 用户态程序打开 /dev/mem，把 MMIO 物理地址 mmap 到自己的虚拟地址空间
int fd = open("/dev/mem", O_RDWR);
void *vaddr = mmap(NULL, 0x1000,
                     PROT_READ | PROT_WRITE,
                     MAP_SHARED,
                     fd,
                     0xFEB00000);  // 物理地址作为 offset

// 现在可以直接读写设备寄存器了
uint32_t val = *(volatile uint32_t*)(vaddr + 0x10);
注意：/dev/mem 在现代系统上通常是受限制的（CONFIG_STRICT_DEVMEM），只能访问有限范围的物理地址。这是安全机制——否则任何用户程序都可以直接篡改硬件。

路径 2：UIO（Userspace I/O 框架）

UIO 是 Linux 内核为"用户态驱动"设计的标准框架。它的核心思路：内核驱动只做最底层的中断处理和 MMIO 映射，把寄存器访问完全交给用户态。

// UIO 设备导出 sysfs 文件：
// /sys/class/uio/uio0/maps/map0/addr   → MMIO 起始物理地址
// /sys/class/uio/uio0/maps/map0/size   → MMIO 区域大小
// /dev/uio0                            → 设备文件

// 用户态通过 mmap 映射 MMIO
int fd = open("/dev/uio0", O_RDWR);
void *regs = mmap(NULL, size, PROT_READ|PROT_WRITE,
                    MAP_SHARED, fd, 0);

// 中断等待：读 /dev/uio0 会阻塞，直到中断发生
uint32_t irq_count;
read(fd, &irq_count, sizeof(irq_count));

DPDK（数据平面开发套件）就使用了类似的思路，把网卡 MMIO 直接映射到用户态，实现零拷贝高速包处理。

路径 3：VFIO（虚拟功能 I/O）

VFIO 更强大：它直接把整个 PCI 设备的BAR 空间、MSI-X 中断、DMA 能力都暴露给用户态，常用于将物理设备直通（passthrough）给虚拟机。

IOMMU 的作用

VFIO 依赖 IOMMU（Intel VT-d / AMD-Vi）。IOMMU 充当 MMU 的角色，但翻译的是设备发出的 DMA 地址（而非 CPU 发出的虚拟地址）。它把设备的 DMA 访问限制在分配给它的物理内存范围内，实现设备隔离和安全 DMA。

典型使用场景

QEMU/KVM 虚拟机中，宿主机用 VFIO 把物理 GPU 直通给虚拟机内的 Windows 游戏使用。虚拟机内的 GPU 驱动就像独占硬件一样操作 BAR 寄存器。

八、完整实例：一个 NVMe SSD 从插上到可用的全链路

假设你买了一块 NVMe SSD，插到主板的 M.2 插槽上（M.2 背后是 PCIe x4 通道）。开机。

阶段一：固件层（BIOS/UEFI）

1 PCIe 链路训练

上电后，PCIe PHY 进行链路协商。检测到有设备插入，链路以 Gen3/Gen4 x4 的速率建立连接。设备开始响应配置空间的读取。

2 BIOS 枚举，读取 NVMe 的配置空间
  • Vendor ID: 0x144D（Samsung）
  • Device ID: 0xA80A
  • Class Code: 0x010802（NVMe 存储控制器）
  • BAR0: 64 位，非可预取，需要 16KB（0x4000）MMIO 空间
3 BIOS 为 BAR0 分配物理地址

BIOS 在 4GB 以上的 MMIO 窗口找到一段空闲的 16KB 空间，分配给 BAR0：物理地址 = 0xF010 0000

写入 BAR0，配置系统代理的地址解码器。至此，CPU 访问 0xF0100000~0xF0103FFF = 读写这个 NVMe 的控制器寄存器。

4 记录到 ACPI MCFG 表

将此设备的 BAR 信息写入 ACPI 表，供操作系统读取。

阶段二：内核启动

5 PCI 子系统初始化

Linux 内核启动后，重新扫描 PCI 总线（也可以直接用 BIOS 的枚举结果）。读取每个 PCI 设备的配置空间，创建 struct pci_dev，记录 BAR 的物理地址和大小。

6 NVMe 驱动匹配并加载 
// 驱动的 PCI ID 表匹配
{ PCI_DEVICE(0x144D, 0xA80A) } // 匹配！

// → nvme_probe() 被调用
struct pci_dev *pdev = ...; // 内核已填好
// pdev->resource[0].start = 0xF0100000 (BAR0 物理地址)
// pdev->resource[0].end   = 0xF0103FFF
7 ioremap 映射 
resource_size_t phys = pci_resource_start(pdev, 0); // 0xF0100000
resource_size_t size = pci_resource_len(pdev, 0);   // 0x4000 (16KB)

// 关键调用：物理 → 虚拟
void __iomem *bar0 = pcim_ioremap(pdev, 0, size);
// bar0 = 0xFFFFC90000100000 (或类似的内核虚拟地址)

// 现在可以访问寄存器了：
//   bar0 + 0x14 → 控制器配置寄存器 (CC.EN 使能位)
//   bar0 + 0x28 → Admin Submission Queue 尾门铃
//   bar0 + 0x30 → Admin Completion Queue 头门铃
8 初始化 NVMe 控制器 
// 1. 写 CC.EN=0 禁用控制器（如果之前被 BIOS 使能）
writel(0, bar0 + 0x14);

// 2. 配置 Admin Queue（设置 AQA, ASQ, ACQ 寄存器）
writeq(admin_sq_phys, bar0 + 0x28); // Admin SQ 基址
writeq(admin_cq_phys, bar0 + 0x30); // Admin CQ 基址

// 3. 写 CC.EN=1 使能控制器
writel(1, bar0 + 0x14);

// 4. 等待 CSTS.RDY=1（控制器就绪）
while (!(readl(bar0 + 0x1C) & 1))
    cpu_relax();

初始化完成后，NVMe 驱动开始用 Admin 命令查询 Namespace 信息、创建 I/O Queue，最终向块层注册一个 /dev/nvme0n1 块设备。

此时的全景：
用户 read(/dev/nvme0n1) → VFS → 文件系统 → 块层 → NVMe 驱动 → 构造 SQ 条目写入 DMA 内存 → writel(tail, bar0 + 门铃偏移) → 这个 writel 是 MMIO 写，最终变成一条 MOV [0xFFFFC90000100030], eax 指令 → MMU 翻译 → 物理地址 0xF0100030 → 系统代理 → PCIe 根端口 → NVMe SSD → 控制器收到门铃写 → 开始 DMA 读取 SQ 条目 → 处理 I/O。

核心要点回顾

1. 物理地址的"多重人格"
同一个物理地址 0xF0100000，可以被路由到 RAM，也可以被路由到设备。谁来决定？系统代理的地址解码器，BIOS 来配置。
2. BAR 是桥梁
BAR 寄存器是设备向 CPU 暴露的接口。"我需要多少地址空间"→ BIOS 分配 → 写入 BAR → 硬件路由生效。
3. ioremap 是翻译官
物理地址不能直接用，通过 ioremap 在内核页表中建立虚拟→物理的映射，并设置 UC（不缓存） 属性。
4. 缓存是大忌
设备寄存器必须设置为 UC/WC。若被缓存，读取到的是过期数据，写入可能被延迟或合并——设备直接失控。
5. 用户态也能玩
通过 /dev/mem、UIO、VFIO 三种方式，用户态进程可以绕过内核直接操作设备寄存器。VFIO + IOMMU 是最安全的方式。
6. 一切自动发生
一旦 ioremap/mmaps 完成，CPU 的每次 MMIO 访问都经由 MMU 自动翻译，TLB 缓存结果。不需要任何软件干预，硬件全自动。

十二、BIOS/UEFI 中的 MMIO 相关配置项

BIOS 是 MMIO 的"总规划师"——它在启动时枚举设备、分配地址、配置路由。以下几项 BIOS 设置直接影响 MMIO 的行为：

📐

12.1 Above 4G Decoding（4GB 以上地址解码）

通常位置：Advanced → PCI Subsystem Settings → Above 4G Decoding

对应理论：第 §2 章"物理地址空间布局"中 TOLUD 和 64 位 BAR 分配。

当此选项启用时：

  • BIOS 可以将 64 位 BAR（如 GPU 的 256MB 显存窗口）分配到 4GB 以上的物理地址
  • 释放 4GB 以下的 MMIO 窗口空间，减少多个设备争抢紧缺的 32 位地址空间
  • 对应第二章的地址分配表：更多大 BAR 出现在"高地址区域"

当禁用时：64 位 BAR 仍然必须在 4GB 以下分配，如果设备多会触发 资源冲突（Code 12）

验证方法：关闭此选项 → 进入 Windows → 打开 msinfo32 → 硬件资源 → 内存 → 观察 GPU 的大 BAR 是否都在 4GB 以下。再开启 → 重新进入 → GPU 的 BAR 地址变为 4GB 以上（如 0x3F80000000 之类）。

注意：Windows 需要此选项开启才能正常使用 Resizable BAR（见下一条）。Linux 通常也需要。
🔧

12.2 Re-Size BAR Support（可变大小 BAR）/ Smart Access Memory（SAM）

通常位置：Advanced → PCI Subsystem Settings → Re-Size BAR Support

对应理论：第 §3.2 章"BAR 寄存器"的 BAR 大小探测机制。

传统上，BAR 的大小由硬件写死——BIOS 只能读出，不能改变。Re-Size BAR 是 PCIe 规范的一个扩展：允许 BIOS/OS 与设备重新协商 BAR 的大小。

  • GPU 典型场景：VRAM 有 8GB，但传统 BAR 只有 256MB → CPU 每次只能访问 256MB 窗口
  • 开启 Re-Size BAR 后：BAR 可以扩展到整个 VRAM（8GB 全部映射）→ CPU 可以一次性访问全部显存
对照 BAR 大小探测（§3.2）：传统方法写 0xFFFFFFFF 返回的结果中"硬化的 0 位"是固定的。Re-Size BAR 让设备固件能够改变这些硬化 0 位的位置，从而改变 BAR 容量。

验证方法：HWiNFO64 → GPU → BAR Size 字段。对比开关前/后的值。HWiNFO64 也会在 GPU 详情页直接显示"Resizable BAR: Enabled/Disabled"状态。
🛡️

12.3 VT-d（Intel）/ AMD-Vi — IOMMU / DMA 重映射

通常位置：Advanced → CPU Configuration → Intel VT-d / AMD SVM

对应理论：第 §7 章"用户态 MMIO 访问"中 VFIO 和 IOMMU 部分。

IOMMU 是设备端的 MMU——它把设备发出的 DMA 地址翻译成物理地址，就像 MMU 翻译 CPU 虚拟地址一样。

  • 禁用时：VFIO 设备直通不可用，虚拟机无法安全使用物理设备。设备 DMA 可以直接访问任意物理内存。
  • 启用时：每个设备的 DMA 被限制在 IOMMU 页表给定的范围内。这是 VFIO、DPDK、SR-IOV 等高级功能的前提。
验证方法：
  • Windows：设备管理器 → 设备属性 → 详细信息 → "DMA 重映射状态" 或 "IOMMU required"
  • Linux：dmesg | grep -i iommu 查看 IOMMU 初始化日志；ls /sys/kernel/iommu_groups/ 查看分组
⚡

12.4 PCI Express Native Control / ASPM

通常位置：Advanced → PCI Subsystem Settings → PCI Express Native Power Management

对应理论：PCIe 电源管理与 MMIO 的关系——当设备进入低功耗状态时，其 MMIO 访问可能会失败或触发挂起恢复流程。ASPM（Active State Power Management）控制 PCIe 链路何时进入 L0s/L1 省电状态。

开启"Native Control"：由 OS 管理 ASPM → 内核驱动可以在空闲时降速链路。对高性能 NVMe 和网卡驱动很重要。

💾

12.5 CSM（Compatibility Support Module）/ Legacy Boot

通常位置：Boot → CSM / Boot Mode → UEFI / Legacy

对应理论：第 §3.4 章"ECAM"和第 §4 章"ACPI 表"。

CSM 启用 = BIOS 模拟传统 BIOS 行为（INT 10h VGA、INT 13h 磁盘等）。这会影响 PCI 枚举的方式：

  • CSM 开：使用传统 PCI 枚举（0xCF8/0xCFC I/O 端口访问配置空间），不使用 ECAM
  • CSM 关（纯 UEFI）：使用 ECAM（MCFG 表中的 MMIO 基址访问配置空间），与现代 PCIe 规范一致
实用建议：要完整验证 §3.4 的 ECAM 机制，确保关闭 CSM。用 RWEverything 查看 MCFG ACPI 表，找到 ECAM 基址，然后直接读该地址 + 偏移 = 访问 PCI 设备的配置空间。

十三、Linux 命令行工具——最精确的 MMIO 验证手段

Linux 的透明性让你能够看到内核视角的每一个细节：物理地址映射、BAR 信息、缓存属性、甚至页表条目。以下工具按信息层级从高到低排列。

13.1 lspci — 查看 BAR 物理地址、大小和类型

lspci -vvv 是 Linux 下的 HWiNFO。它直接读取 sysfs 中内核解析过的 PCI 配置空间信息。

 lspci -vvv — NVMe 设备输出示例
$ lspci -vvv -s 02:00.0 02:00.0 Non-Volatile memory controller: Samsung Electronics Co Ltd NVMe SSD Controller Subsystem: Samsung Electronics Co Ltd Device a801 ... Region 0: Memory at fe800000 (64-bit, non-prefetchable) [size=16K] ↑ BAR0：物理地址 = fe800000，64 位，不可预取，16KB 对应理论：§3.2 (BAR 类型 + 大小) + §8.2 步骤 3 (物理地址分配) Region 4: Memory at fe804000 (32-bit, non-prefetchable) [size=4K] ↑ BAR4：第二个寄存器窗口 Capabilities: [50] MSI-X: Enable+ Count=33 Masked- ↑ MSI-X 中断（对应 §7 用户态访问中断） Capabilities: [100] Advanced Error Reporting Capabilities: [200] Secondary PCI Express LnkCap: Port #0, Speed 16GT/s, Width x4 ↑ PCIe 4.0 x4 链路（对应 §8.1 步骤 1 链路训练） $ lspci -vvv | grep -E "Region|Memory" # 只看 BAR 信息 Region 0: Memory at e0000000 (64-bit, prefetchable) [size=256M] Region 2: Memory at f0000000 (64-bit, non-prefetchable) [size=32M] ↑ NVIDIA GPU：两个 BAR，显存可预取，寄存器不可预取

"Region 0" = BAR0，"Region 1" = BAR1，以此类推。"Memory at 0xXXXX" 表示 Memory Space 类型，"I/O ports at 0xXXXX" 表示 I/O Space 类型。

如果在 WSL2 中：WSL2 默认不能直接访问 PCI 设备（内核是虚拟化的）。需要物理机上的 Linux，或者使用 Windows 下的 HWiNFO64 代替。

13.2 /proc/iomem — 内核的"物理地址空间全景图"

/proc/iomem 是最重要的一个文件——它以文本形式展示了内核所知的全部物理地址分配。这本质上就是 msinfo32"硬件资源→内存"的 Linux 版本，但信息更丰富。

 cat /proc/iomem（节选，带注释）
$ cat /proc/iomem 00000000-00000fff : Reserved ← BIOS 保留 00001000-0009ffff : System RAM ← 真正的 RAM 000a0000-000bffff : PCI Bus 0000:00 ← VGA 显存窗口 legacy 000c0000-000cffff : PCI Bus 0000:00 ← BIOS ROM 映射 00100000-3fffffff : System RAM ← 主内存（到 1GB） 40000000-403fffff : PCI Bus 0000:00 ← 32 位 MMIO 窗口开始 e0000000-efffffff : PCI Bus 0000:01 e0000000-efffffff : 0000:01:00.0 ← GPU BAR0 (256MB, WC) f0000000-f1ffffff : 0000:01:00.0 ← GPU BAR2 (32MB, UC) fe800000-fe803fff : 0000:02:00.0 ← NVMe BAR0 (16KB, UC) fec00000-fecfffff : PNP0003:00 ← I/O APIC (UC 强不缓存) fed00000-fed00fff : PNP0103:00 ← HPET 100000000-41f37ffff : System RAM ← 4GB 以上系统内存 39f0000000-3a0fffffff : PCI Bus 0000:01 39f0000000-3a0fffffff : 0000:01:00.0 ← GPU ReBAR (8GB 全显存映射!) // 对照 §2 的地址划分表： // ✓ 传统区域 (0xA0000-0xBFFFF = VGA, 0xC0000-0xFFFFF = BIOS) // ✓ TOLUD 大概在 0x40000000（1GB），之上全是 PCI MMIO // ✓ 4GB 以上有系统 RAM，也有大 BAR（如 GPU ReBAR 在 ~0x3A00000000） // ✓ 固定硬件 (APIC, HPET) 有固定的保留地址
/proc/iomem 的缩进层级 = PCI 拓扑层级
缩进越深，设备在 PCI 树中越深。比如 PCI Bus 0000:01 → 0000:01:00.0 表示设备挂在 Bus 1 上。这与第三章"PCI 枚举"中提到的 Bus/Device/Function 层级直接对应。

13.3 /proc/mtrr — 验证 MTRR 缓存策略

这个文件直接展示 CPU 的 MTRR 寄存器内容，让你验证 §5.4 的缓存控制理论。

 cat /proc/mtrr
$ cat /proc/mtrr reg00: base=0x000000000 ( 0MB), size=32768MB: write-back, count=1 ↑ 低 32GB RAM 设为写回（WB，性能最优的缓存策略） reg01: base=0x0e0000000 (3584MB), size= 256MB: write-combining, count=2 ↑ 显存 BAR0 区域设为写合并（WC）——对应 §5.4 PAT 表 WC 类型 reg02: base=0x0fe800000 (4072MB), size= 8MB: uncachable, count=1 ↑ MMIO 设备寄存器区域设为不可缓存（UC）——对应 §5.4 PAT 表 UC 类型 // 三条 MTRR 记录完美验证了 §5.4 的结论： // RAM → WB（写回，带缓存） // GPU 显存 BAR → WC（写合并，部分缓存优化） // NVMe/网卡寄存器 BAR → UC（强不缓存，直接到达设备）
注意：MTRR 在现代系统中逐渐被 PAT 替代。Intel 从 Haswell 起、AMD 从 Zen 起，推荐使用 PAT 做精细控制。你可能会看到 /proc/mtrr 只有少量条目，但实际的缓存类型由 PAT + MTRR 组合决定（取最严格值）。

  4. 查缓存策略：cat /proc/mtrr 或 RWEverything → 找到 UC 和 WC 条目 → 对照 §5.4 缓存类型表
